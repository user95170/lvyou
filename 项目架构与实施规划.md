## 1. 项目总体目标与范围

- 面向对象
  - 不同地域、年龄、性别等人群的游客
  - 内蒙古自治区内的文旅管理部门和景区运营方
- 核心目标
  - 基于多源异构数据构建内蒙古旅游知识库
  - 对不同游客群体进行画像与需求预测
  - 提供覆蓋吃、住、行、游的一体化个性化推荐
  - 结合实时交通生成和动态调整旅游路线与出行指南
- 非功能目标
  - 可扩展：后续可扩展到其他省份或其他文旅场景
  - 可维护：采用清晰分层和模块化设计，便于迭代
  - 可复用：算法与数据模块尽量解耦，便于在论文或其他项目中复用

### 1.3 新需求对齐（自然语言 + 画像 + 个性化电子地图）

- 目标对齐：
  - 自然语言交互：面向“出行时间、预算、交通方式、兴趣偏好”等输入，完成意图与槽位抽取；
  - 用户画像深化：显式纳入性别、年龄、地域及其交叉特征；与行为特征合并为“用户侧特征”；
  - 个性化电子地图：融合景点/美食/住宿/交通/活动/特产多要素，按用户画像加权排序、筛选与展示；
  - 行程与路线：按画像与约束生成多日行程，提供“最快/少换乘/步行优先”等路线方案；
- 落地产出：
  - 后端 API：NLU 解析、个性化地图检索、行程建议、路线方案；
  - 数据：画像字段扩展（性别/年龄段/地域）、行程与路线草案表；
  - 前端：地图页与详情页增强（适配提示）、行程规划页；
- 文档协同：接口与模块设计、数据库设计在 docs 目录中同步更新，作为本文档的实现支撑。

## 2. 系统总体架构

### 2.1 技术选型概览

- 后端：Python + Flask
- 前端：Vue.js
- 数据库：SQLite（本地默认）/ MySQL（可选），Redis（缓存与会话）
- 地图与路线：高德/百度地图开放接口
- 其他：
  - 爬虫与数据处理：Requests、Scrapy 或自写爬虫脚本，配合定时任务
  - 文本分析：结巴分词、常见 NLP 工具包等
  - 机器学习与深度学习：Scikit-Learn、XGBoost、PyTorch 或 TensorFlow 等

### 2.2 分层架构

自上而下分为四层：

1. 表示层（前端展示与交互）
   - 用户端 Web：推荐内容展示、地图与路线、行程管理、评价反馈
   - 管理后台 Web：景点与资源管理、数据看板、模型与任务监控

2. 应用服务层（业务逻辑与接口）
   - 用户与认证服务
   - 推荐与行程服务
   - 路线规划与地图服务（封装地图 API）
   - 评价与反馈服务
   - 管理后台服务

3. 领域与算法层（画像与模型）
   - 用户画像与用户分群模块
   - 推荐算法模块（协同过滤、内容推荐、混合推荐）
   - 行为预测与时间序列模块（可选）
   - 规则引擎（节假日、天气、民族文化偏好等规则）

4. 数据与基础设施层
   - 数据库（SQLite/MySQL）：用户、景点、行程、评价、行为日志、画像等
   - Redis：热门数据缓存、会话、配置
   - 文件与模型存储：原始爬虫数据、清洗后数据集、训练好的模型
   - 外部服务：地图、交通、社交、电商、OTA 等开放接口

### 2.3 关键子系统与模块

- 数据采集与治理子系统
  - 多源数据采集（社交媒体、电商、OTA、生活服务、交通、官方数据）
  - 数据清洗与标准化（去重、缺失值处理、编码统一等）
  - 文本处理（分词、主题识别、情感分析、口碑评分）

- 用户画像与行为分析子系统
  - 用户基础信息与偏好采集
  - 用户行为数据汇总（点击、收藏、下单、评价、停留时间等）
  - 聚类分析、关联规则挖掘，形成用户分群与标签体系

- 推荐与预测子系统
  - 基础推荐：规则推荐、热门榜、地域+季节推荐
  - 协同过滤与内容推荐：基于用户-项目行为和项目内容特征
  - 混合推荐：将多种算法结果加权融合
  - 行为预测与时间序列（视时间和课题深度可选）

- 路线规划与出行指南子系统
  - 地图与路线查询封装
  - 根据推荐结果和用户约束生成多日行程（吃住行游）
  - 结合实时交通数据进行行程动态调整

- 个性化电子地图与详情子系统（新增细化）
  - 地图要素：景点/美食/住宿/交通节点/活动/特产多层展示；
  - 排序与筛选：结合画像（性别/年龄/地域）与多源评分，执行加权重排与过滤；
  - 详情页：按要素类型展示“基础信息 + 评分/评论 + 画像适配说明（如偏辣/安静/夜生活等）”；

- 自然语言理解与意图识别子系统（新增）
  - 能力：意图分类 + 槽位抽取（目的地、时间、预算、人数、出行方式、偏好等）；
  - 输出：标准化查询 JSON，直连个性化地图与行程服务；
  - 技术：规则 + 轻量 NLP 组件（可用结巴/简单词典 + 正则；后续可替换为更强 NLU）。

- Web 应用与接口子系统
  - 统一 RESTful API（用户、推荐、行程、评价、管理等）
  - 前端页面路由与组件（用户端 + 管理端）

### 2.4 数据流与运行模式

- 离线数据流
  - 爬虫/API 抽取 → 清洗与标准化 → 特征构造 → 训练集/测试集 → 模型训练与评估 → 模型文件与画像写入

- 在线服务流
  - 用户请求 → 读取画像与行为 → 构造实时特征 → 调用推荐与路线服务 → 过滤与排序 → 返回结果并记录行为日志

## 4. 算法与推荐模型规划

### 4.1 阶段性算法路线

1. 第一阶段：可用为主的基线方案
   - 基于规则和统计的热门推荐（按季节、地区、节假日）
   - 简单的标签匹配推荐（用户偏好标签对景点标签）

2. 第二阶段：协同过滤与内容推荐
   - 用户-项目行为矩阵构建
   - 基于用户或项目的协同过滤
   - 基于景点、商品内容特征的内容推荐

3. 第三阶段：混合推荐与排序模型
   - 将规则、协同过滤、内容推荐结果做加权融合
   - 引入学习排序或点击率预测模型（如 XGBoost）进行最终排序

4. 第四阶段（可选）：时间序列与需求预测
   - 使用时间序列或深度模型预测景点热度与客流
   - 结合预测结果对推荐结果与路线规划进行调整

### 4.2 关键评估指标

- 推荐准确性：点击率、收藏率、下单转化率等
- 覆盖率与多样性：是否过于集中在少数热门景点
- 用户满意度：问卷与在线评价
- 系统性能：响应时间与可用性

### 5.5 当前进展与下一阶段重点（当前版本）

- **当前已完成的关键实现（推荐相关）**  
  - 景点推荐接口 `/api/recommend/scenic-spots`：
    - 支持热门、多源融合排序与画像重排（popular / profile / multi_source）；
    - 支持基于 item-based CF 的个性化推荐（strategy=`cf`），并在 `content_standard` 中维护相似度矩阵；
    - 支持基于 Biased MF 的矩阵分解推荐（strategy=`mf`），在在线请求时按评分表训练轻量模型；
    - 支持 Hybrid 混合策略（strategy=`hybrid`），在同一候选集合上融合多源基线分、CF 相似度得分和 MF 预测得分（默认权重约为 0.2 / 0.4 / 0.4）；

## 6. 代码与目录结构建议

> 目录结构可根据实际开发工具略作调整，但建议自始至终保持统一，避免后期整理困难。

建议在項目根目录下采用类似结构：

- backend
  - app（Flask 应用主代码，按模块划分包）
  - tests（后端测试脚本）
  - requirements.txt
- frontend
  - src（Vue.js 应用主代码，按模块划分包）
  - tests（前端测试脚本）
  - package.json
- data
  - raw（原始数据存储）
  - processed（清洗后数据存储）
- docs
  - 项目文档（本文档、接口文档、数据库设计文档等）
- models
  - 推荐模型、画像模型等的保存文件

当前本文档建议位置：项目根目录或 docs 目录中，以本文为架构与实施规划唯一基准文档。

## 7. 风险与后续扩展建议

- 主要风险
  - 多源数据获取难度和接口限制
  - 时间有限下算法与系统功能的取舍
  - 爬虫和接口变更带来的维护成本

- 建议应对
  - 优先确保一到两个数据源完整贯通，再逐步扩展
  - 在时间不足时，优先保证端到端可用流程，再做算法深度优化
  - 对关键脚本和配置做好注释与文档

- 后续扩展方向
  - 扩展到更多地区和跨区域旅游场景
  - 引入更多实时数据（天气、票务、舆情等）
  - 使用更先进的推荐与大模型技术提升智能化程度
