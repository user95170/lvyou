# 数据采集与处理方案

> 版本：v1.0（多源数据设计版）  
> 范围：围绕项目书中提出的多源数据（社交媒体、电商/OTA、地图/交通、官方统计等），给出**可落地的采集与处理架构**，并说明在当前原型中已经实现或可直接实现的部分。

---

## 1. 多源数据总体架构

### 1.1 目标与约束

- 目标：
  - 将社交媒体热度、电商/OTA 行为数据、地图/交通信息、官方统计等多源数据，整合为可支撑**热门分析、个性化推荐、行程规划与决策支持**的统一数据层。
- 约束：
  - 外部平台多需申请 API Key 或登录态，不适合在课堂环境中大规模真实抓取；
  - 本项目以**“清晰架构 + 可运行示例 + 可扩展代码骨架”**为主，便于后续在合法合规前提下扩展真实数据采集。

### 1.2 分层设计

- **数据源层**：
  - 社交媒体：微博、短视频平台（抖音/快手）、小红书等；
  - OTA/本地生活：携程、去哪儿、美团、大众点评等；
  - 地图/交通：高德地图、百度地图、12306 等；
  - 官方与研究：文旅厅官网、统计局、CNKI 等。
- **采集与落地层（pipelines）**：
  - 以 `app/pipelines/` 目录下的脚本为主，每类数据源提供独立模块；
- 统一对接应用工厂 `create_app()`，通过 SQLAlchemy 写入 SQLite（默认）或 MySQL；
  - 对敏感接口统一通过环境变量注入 Key，不在仓库中硬编码。
- **标准化与融合层**：
  - 将外部数据规整为项目内部统一结构：
    - 景点、酒店、美食 → 映射到 `scenic_spot` / `hotel` / `food_place` 或 `content_standard`；
    - 舆情热度、口碑评分、销量等 → 写入 `content_standard`，作为 `popularity_score`、`sentiment_score` 等特征；
  - 后续推荐算法、行程规划等模块只依赖内部标准化表，而不关心外部数据细节。

### 1.3 标准化字段与枚举约定（权威口径）

- `content_standard` 表用于存放外部来源的“可复用特征汇总”（评分、评论数、热度、情感等），供推荐/画像/聚合脚本统一读取。
- 唯一性口径：`entity_type + entity_id + source_type` 视为同一条外部特征记录，重复导入以更新（upsert）方式覆盖，并刷新 `last_update`。

- `entity_type` 取值：
  - `scenic_spot`：景点（对应表 `scenic_spot`）
  - `hotel`：酒店（对应表 `hotel`）
  - `food_place`：美食地点（对应表 `food_place`）
  - 兼容说明：历史数据可能存在 `food`，导入脚本会归一为 `food_place`，推荐模块也兼容读取 `food`。

- `source_type` 取值：
  - `internal_rating`：内部评分聚合（由 `aggregate_content.py` 写入）
  - `ota_stats`：OTA/本地生活平台统计（由 `load_ota_data.py` 或 `load_external_features.py` 写入）
  - `social_media`：社交热度/舆情聚合（由 `load_external_features.py`、`fetch_google_trends.py`、`fetch_wikipedia_pageviews.py` 等写入）
  - 兼容说明：历史数据可能存在 `ota`，导入脚本会迁移为 `ota_stats`，推荐模块也兼容读取 `ota`。

- `summary`（JSON）字段约定：
  - 当 `source_type = ota_stats`：
    - `platform`：来源平台（携程/美团/大众点评等）
    - `external_rating` / `rating`：评分（写入时会同时写入以兼容旧模块）
    - `review_count` / `external_review_count`：评论数（写入时会同时写入以兼容旧模块）
    - `price_range`：价格区间（字符串）
    - `tags`：标签数组
  - 当 `source_type = social_media`：
    - `interaction_sum`：互动量/热度（整数）
    - `sentiment_avg`：平均情感（浮点，范围建议 0~1）

### 1.4 合规与隐私（必须遵守）

- 合法来源优先：优先使用官方 API、公开数据集或人工整理数据；不建议在课堂/演示环境进行大规模爬取。
- 密钥不入库不入仓：所有 Key 仅通过环境变量注入（如 `AMAP_API_KEY`、`AMAP_WEB_KEY`），仓库内不硬编码。
- 最小化采集原则：对社交平台内容仅保留聚合统计（数量、互动、情感等），不落个人身份信息（昵称、ID、联系方式等）。
- 速率限制与礼仪：遵守平台 QPS/并发限制，必要时使用缓存与退避重试，避免对外部服务造成压力。

## 2. 现有内部数据与聚合/导入脚本

当前项目已经具备的内部数据与聚合能力：

- **基础表**：`scenic_spot`、`hotel`、`food_place`、`user`、`rating`、`user_profile`、`content_standard` 等；
- **评分导入与聚合脚本**：
  - `aggregate_content.py`：基于评分计算各类内容的 `popularity_score`；
  - `aggregate_user_profile.py`：根据评分行为聚合用户画像；
  - `aggregate_scenic_cf.py`：基于评分构建景点之间的协同过滤相似度；
  - `import_ratings_from_csv.py`：从线下整理的真实评分 CSV 批量导入 `rating` 表，并更新景点/酒店/美食的 `rating_avg` 与 `rating_count` 聚合字段，是面向真实打分数据的正式导入通道。
- 这些脚本构成了“**内部数据管道**”，后续多源数据采集的输出也将统一写入这些表或新增扩展表。

### 2.1 刷新策略与调度（建议）

- 内部评分与画像：当导入/产生新评分后，建议按顺序运行：
  - `import_ratings_from_csv.py` → `aggregate_content.py` → `aggregate_user_profile.py`
- AMap POI 基础数据：首次部署或新增城市时运行 `load_amap_data.py`；后续可按月/按需补采。
- OTA 数据：当 `ota_sample.csv` 更新或新增平台数据时运行 `load_ota_data.py`；脚本会 upsert 并更新 `last_update`。
- 社交热度：Google Trends / Wikipedia 等外部信号建议按周或按月更新；更新后可运行 `aggregate_content.py` 使热度参与综合人气。

### 2.2 数据质量校验与可观测性（落地）

- 字段与类型校验：导入阶段对评分/评论数等字段做数值化校验（非数值则置 0 或跳过），避免污染特征。
- 去重与幂等：`content_standard` 以 `entity_type + entity_id + source_type` 做 upsert；重复运行脚本不应产生重复记录。
- 匹配质量：`DataMatcher` 会输出匹配成功/失败统计；对未匹配项建议人工修正名称/城市后重跑导入。
- 覆盖率与新鲜度：可运行 `analyze_feature_coverage.py` 输出各实体类型 OTA/Social 覆盖率；并以 `content_standard.last_update` 评估数据新鲜度。

## 3. 社交媒体与 UGC 数据

### 3.1 数据来源与用途

- 微博：旅游相关话题、景点关键词的微博数、转发量、点赞量等，用于估计**网络关注度**；
- 小红书 / 短视频平台：与景点/路线相关的笔记/视频数量、互动数，用于衡量**内容传播热度与口碑**；
- 这些数据可以作为：
  - `content_standard.popularity_score` 的补充维度；
  - 用户画像中的“关注热点”或“兴趣话题”特征。

### 3.2 采集架构设计

- 采集模块：`app/pipelines/social_media_fetcher.py`（规划中，可按需要扩展）：
  - 主要职责：
    - 根据关键词列表（景点名、城市名、主题词）调用各平台公开 API 或模拟搜索接口；
    - 解析返回 JSON/HTML，提取：发布时间、互动数、正文摘要、关联景点等；
    - 聚合到景点级别：例如统计某景点在过去一周的相关内容数与互动总量。
- 建议落地方式：
  - 若不新增表，可直接汇总到 `content_standard`：
    - `entity_type = "scenic_spot"`
    - `source_type = "social_media"`
    - `summary` 中存放：近 7 天/30 天微博数、平均情感得分、互动量等；
  - 若后续需要细粒度分析，可扩展新表 `social_media_post`，再做二次聚合。

### 3.3 当前原型可执行的部分

考虑到课堂环境下不宜直接调用真实平台 API，本项目原型阶段：

- 在代码层面提供**采集骨架**（请求、解析、落地的流程），API Key、具体 URL 由用户自行配置；
- 在文档层面给出可行方案，便于后续在合规前提下扩展真实采集逻辑。

### 3.4 Wikipedia Pageviews 现状与限制（重要）

- 目的：尝试以 Wikipedia Pageviews 作为“社交热度”弱信号（`interaction_sum`）.
- 已实现：
  - 多实体导入脚本 `fetch_wikipedia_pageviews.py`，支持 `scenic_spot/hotel/food_place/all`；
  - 检索增强：`intitle` 精确检索→普通检索→Wikidata 回退（sitelinks）→品牌级回退；
  - 过滤与回退：P31 类型过滤（hotel=Q27686, restaurant=Q11707）、地理过滤（P17=中国 或 P131 匹配城市/区县）、Pageviews 结束日期置为“昨天（UTC）”，user 无数据时回退 all-agents；
  - 调试计数：输出 zh/en 标题/Wikidata/品牌命中与 views>0 计数.
- 覆盖结论（当前数据）：
  - 酒店/餐饮实体在 Wikipedia 的覆盖接近 0；即使加上 Wikidata+品牌+地理回退，views>0 仍无命中；
  - 景点侧在本数据集的命名与地名粒度也难以稳定命中，调试导入（2000 条、90 天窗口）统计仍为 0.
- 建议：
  - 将 Wikipedia 用作“景点侧的可选补充”，不作为酒店/餐饮社交热度来源；
  - 优先引入更贴近本地生活的 UGC/社交平台数据源：微博、小红书、美团/大众点评（话题量、笔记量、互动量、评分/评论数等）；
  - 若后续需要品牌层面的热度，可按品牌级（而非门店级）聚合外部信号，并在 `summary` 中显式标注 `brand_based=true` 以便审核与调权.
- 适用性与局限（补充说明）：
  - 覆盖偏差：酒店/餐饮等本地实体往往不满足 Wikipedia“显著性/知名度”准则（尤其是 Local interests 与 Restaurants 类目），因此条目稀缺，导致 Pageviews 在此类实体上缺乏代表性与覆盖；
  - 事件驱动峰值：Pageviews 易受突发新闻或热点事件驱动产生短时峰值，未必能反映持续的消费或到访热度（应与 Trends 等时序信号互证）；
  - 交叉核验：建议仅在“景点/地标类目标”上做弱特征参考，并与 Google Trends、微博/小红书互动量等信号做一致性比对再决定是否入模；
  - 参考链接：
  - [Wikipedia Pageview statistics](https://en.wikipedia.org/wiki/Wikipedia:Pageview_statistics)
  - [Notability（Local interests）](https://en.wikipedia.org/wiki/Wikipedia:Notability_(local_interests))
  - [Notability（Restaurants）](https://en.wikipedia.org/wiki/Wikipedia:Notability_(restaurants))

### 3.5 Wikipedia Pageviews API 用法与参数（补充）

- 端点（per-article）：
  - `https://wikimedia.org/api/rest_v1/metrics/pageviews/per-article/{project}/{access}/{agent}/{article}/{granularity}/{start}/{end}`
- 关键参数：
  - `project`：如 `zh.wikipedia.org`、`en.wikipedia.org`；
  - `access`：`all-access | desktop | mobile-web | mobile-app`；
  - `agent`：`all-agents | user | spider`（人类访问通常选 `user`，无数据时可回退 `all-agents`）；
  - `article`：词条标题（需 URL 编码，空格常用下划线 `_`，区分大小写，建议使用站点规范标题）；
  - `granularity`：`daily | monthly`；
  - `start`/`end`：时间范围，`YYYYMMDD`（如 `daily/20231001/20231031`）。
- curl 示例：

  ```bash
  curl "https://wikimedia.org/api/rest_v1/metrics/pageviews/per-article/zh.wikipedia.org/all-access/user/呼和浩特/daily/20231001/20231031"
  ```

- 返回：`items` 为按 `granularity` 聚合的时序数组（`timestamp, views` 等）。标题不存在或无数据可能返回 `404` 或空 `items`；应结合多语种与同义标题做回退。
- 参考：
  - Getting started（入口与通用说明）：[docs](https://doc.wikimedia.org/generated-data-platform/aqs/analytics-api/documentation/getting-started.html)
  - Page metrics 示例（per-article）：[docs](https://doc.wikimedia.org/generated-data-platform/aqs/analytics-api/examples/page-metrics.html)
  - Page views 参考（概念与口径）：[docs](https://doc.wikimedia.org/generated-data-platform/aqs/analytics-api/concepts/page-views.html)

### 3.6 Wikimedia API 使用规范（附）

- **User-Agent 必须规范**：使用可识别的 UA，包含工具名/版本与联系方式。
  - 示例：`User-Agent: TourismMap/1.0 (admin@example.com) requests/2.32`
  - 浏览器环境无法自定义 UA 时，使用 `Api-User-Agent` 头部。
  - 参考：[API:Etiquette](https://www.mediawiki.org/wiki/API:Etiquette)、[User-Agent policy](https://foundation.wikimedia.org/wiki/Policy:User-Agent_policy)
- **请求速率与并发**：
  - 建议串行发送或小批次请求，避免高并发；优先用“管道符”等批量参数一次取回多个条目。
  - 命中速率限制（返回 `ratelimited` 等错误）时，采用指数退避重试。
- **压缩与带宽**：设置 `Accept-Encoding: gzip`。
- **maxlag 参数（非互动任务）**：为非交互任务设置 `maxlag=<秒>`，在集群压力大时自动退让，降低对服务的影响。
- **示例（Python requests）**：

```python
headers = {
  "User-Agent": "TourismMap/1.0 (admin@example.com) requests/2.32",
  "Accept": "application/json",
  "Accept-Encoding": "gzip",
}
params = {
  "maxlag": 5,
}
# requests.get(url, headers=headers, params=params, timeout=(20, 60))
```


### 3.7 Wikimedia Analytics 其他指标（edits/editors）（附）

- **端点（per-page）**：
  - Edits（某词条在一段时间内的编辑次数）：`/metrics/edits/per-page/{project}/{page-title}/{editor-type}/{granularity}/{start}/{end}`
  - Editors（某词条在一段时间内的编辑者数量）：`/metrics/editors/per-page/{project}/{page-title}/{editor-type}/{granularity}/{start}/{end}`
- **关键参数**：
  - `project`：如 `en.wikipedia.org`、`zh.wikipedia.org`
  - `editor-type`：`all-editor | user | bot`
  - `granularity`：`daily | monthly`
  - `start`/`end`：`YYYYMMDD`
- **curl 示例（edits）**：

  ```bash
  curl "https://wikimedia.org/api/rest_v1/metrics/edits/per-page/en.wikipedia.org/Jupiter/user/daily/20231001/20231031"
  ```

- **curl 示例（editors）**：

  ```bash
  curl "https://wikimedia.org/api/rest_v1/metrics/editors/per-page/en.wikipedia.org/Jupiter/all-editor/daily/20231001/20231031"
  ```

- **注意**：
  - 返回数据中可能省略 0（无编辑的日期会被跳过），对齐时间序列时需补零（参考“Compare page metrics”教程）。
  - 其余礼仪与 Header 与 3.6 相同（规范 UA、gzip、必要时 backoff）。
- **参考**：
  - 示例与教程：[Page metrics examples](https://doc.wikimedia.org/generated-data-platform/aqs/analytics-api/examples/page-metrics.html)、[Compare page metrics](https://doc.wikimedia.org/generated-data-platform/aqs/analytics-api/tutorials/compare-page-metrics.html)
  - API 入口：[Analytics API](https://doc.wikimedia.org/generated-data-platform/aqs/analytics-api/)

## 4. OTA、酒店预订与本地生活数据

### 4.1 数据来源

- 携程、去哪儿：景点门票销量、酒店预订量、评价评分；
- 美团、大众点评：餐饮、住宿的用户评价数、人均价格、评分分布；
- 电商平台（淘宝等）：当地特产销售数据，辅助判断某些物产或景区的热度.

### 4.2 采集与整合方案（当前实现）

- 数据文件：`backend/data/ota_sample.csv`
  - 由人工从携程、美团、去哪儿、大众点评等平台整理的 70 条真实 OTA 数据；
  - 字段：`entity_type, entity_name, city, source_platform, external_rating, external_review_count, price_range, tags, description`.
- 采集模块：`app/pipelines/collectors/ota_collector.py`（`OTACollector`）
  - 支持从 CSV 读取并解析 OTA 数据；
  - 兼容旧列名（`type, name, platform, rating, review_count`）与新列名；
  - 为后续扩展真实 OTA API 采集预留接口（`source="api"`）.
- ETL 脚本：`app/pipelines/load_ota_data.py`
  - 通过 `OTACollector` 采集数据；
  - 调用 `DataMatcher`（`processors/data_matcher.py`）按名称+城市做实体匹配；
  - 将外部评分、评论数、价格区间等指标写入 `content_standard`：
    - `entity_type = scenic_spot | hotel | food_place`;
    - `source_type = "ota_stats"`（兼容历史 `source_type = "ota"`）；
    - `summary` 中存放 `platform`、`external_rating/rating`、`review_count/external_review_count`、`price_range`、`tags` 等字段.

### 4.3 当前原型实现现状

- 在没有开放 OTA API 的前提下，本项目已经：
  - 基于真实 OTA 页面人工整理了一份 `ota_sample.csv`；
  - 通过 `OTACollector + load_ota_data.py` 验证了完整的“CSV → 匹配 → 融合 → 入库”流程；
  - 通过 `DataMatcher` 将 OTA 数据匹配到内部实体（景点/酒店/美食）的 ID，并以 `source_type = ota_stats` 写入 `content_standard`；该表与高德 POI 基础表（`scenic_spot/hotel/food_place`）通过 `entity_type/entity_id` 关联。
- 后续若接入真实 OTA API，只需在 `OTACollector._fetch_from_api` 中补充实际请求逻辑，并复用现有匹配与落地流程.

## 5. 地图与交通数据

### 5.1 地图与路线规划

- 已实现部分：
  - 行程规划接口 `POST /api/route/plan` 使用景点经纬度 + 最近邻启发式规划游览顺序；
  - 配置环境变量 `AMAP_WEB_KEY` 后，可调用高德驾车路径规划 API 获取更精确的总路程.
- AMap 参考与接口：
  - 文档总览（路径规划）：[指南](https://lbs.amap.com/api/webservice/guide/api/direction)
  - 驾车：`/v3/direction/driving`；公交（综合换乘）：`/v3/direction/transit/integrated`；步行：`/v3/direction/walking`；距离测量：`/v3/distance`
  - 路径规划 2.0（v5）：[新路线规划文档](https://lbs.amap.com/api/webservice/guide/api/newroute)
    - 公交（`/v5/direction/transit/integrated`）参数要点：`city1/city2` 必填（仅 `citycode`，同城/跨城由两者是否相同决定）；可选 `ad1/ad2`（仅 `adcode`，用于限定起终点所在行政区，见[行政区划查询](https://lbs.amap.com/api/webservice/guide/api/district)）。
    - 驾车（`/v5/direction/driving`）支持 `waypoints` 多途经点，使用分号 `;` 分隔，最多 16 个。
  - 物流距离测量矩阵（1-N）：[Distance API](https://lbs.amap.com/api/webservice/guide/logistics-api/distance)
  - 逆地理编码（v3）：[geocode/regeo](https://lbs.amap.com/api/webservice/guide/api/georegeo)
  - 行政区划查询（v3）：[config/district](https://lbs.amap.com/api/webservice/guide/api/district)
- 使用方式：
  - 持有 Web 服务 Key 即可 REST 调用，失败时回退为 Haversine 估算（系统已在 `/api/route/plan` 中内置回退）.
- 数据利用方式：
  - 未来可根据不同游玩时间、路网拥堵情况，对路线进行再优化；
  - 结合交通数据（高峰/低谷时段）给出“推荐出行时间段”.

### 5.2 交通客流与出行频次

- 数据来源（设计）：
  - 通过高德/百度交通 API 查询部分干道、景区周边的拥堵指数；
  - 通过 12306 或民航数据（若有公开接口/数据集）估计节假日客流量；
- 整合方式：
  - 将“节假日客流指数”、“实时拥堵指数”等指标写入 `content_standard`，作为 `entity_type = "scenic_spot"` 的附加特征；
  - 在推荐时，可以对“人特别多”的景点做轻微降权，给出更舒适的替代方案（可在后续版本中增加）.

## 7. 与推荐与画像模块的衔接

多源数据采集完成后，最终都要落到推荐与画像模块可使用的特征上：

- 内容相关特征：

#### 8.1.1 Google Trends 代理与排错

- 建议代理参数：`--https-proxy http://127.0.0.1:50287`（优先使用 http:// 协议；必要时再试 https:// 或 socks5h://）。
- 也可通过环境变量方式临时设置（PowerShell）：

  ```powershell
  $env:HTTPS_PROXY="http://127.0.0.1:50287"
  $env:HTTP_PROXY="http://127.0.0.1:50287"
  $env:NO_PROXY="localhost,127.0.0.1"
  python -m app.pipelines.fetch_google_trends --geo CN --window "today 3-m" --max 120 --sleep 2.5 --out E:\旅游\data\google_trends_social.csv
  ```

- 脚本内置的网络鲁棒性：
  - 先尝试 DIRECT，再按需尝试代理（当仅给出无协议的地址时，会自动尝试 `http://`、`https://`、`socks5h://` 三种）。
  - 初始化前设置 `HTTP(S)_PROXY/NO_PROXY` 环境变量，确保 pytrends 的首次 Cookie 请求走代理。
  - `requests_args={"verify": false}` 以容忍某些代理的证书问题；`timeout=(20,60) + retries/backoff` 提升稳定性.

- 常见错误与处理：
  - `ConnectTimeout to trends.google.com`：检查 VPN 已开启、代理端口正确、网络未被公司防火墙拦截；可增加 `--sleep` 并重试.
  - `TypeError: expected string or bytes-like object, got 'dict'`（旧版脚本）→ 已修复：脚本传入 `TrendReq.proxies` 为 `list[str]`（pytrends 期望可做列表操作）。
  - `Failed to resolve 's'`（代理地址解析异常）→ 使用完整协议（如 `http://127.0.0.1:50287`），避免将 dict 误传为 proxies.
  - 使用 `socks5h://` 时需安装 `pysocks`：`pip install pysocks`.

- 抓取完成后按“导入流程”三步执行导入→聚合→评估.

#### 8.1.2 Google Trends 时间窗口与关键词上限（补充）

- 关键词上限：每次请求最多 5 个关键词（`kw_list`）；可分批请求并合并结果；
- 时间窗口：
  - `today 12-m` 等“月级窗口”支持 1/3/12 个月等预设；
  - `now 7-d` 等“日/小时级”窗口仅支持 1 天或 7 天等预设；
  - 也可使用 `YYYY-MM-DD YYYY-MM-DD` 指定区间；
- 建议：
  - 结合 `now 7-d` 与 `today 12-m` 做近期优先融合，必要时在 `now 7-d` 上做“半衰期衰减”；
  - 分批构造关键词（景点名/城市/区县组合），对每批取和或最大值以缓和分词/歧义影响；
- 参考：pytrends 文档（Common API parameters / Interest Over Time）：[PyPI](https://pypi.org/project/pytrends/)

#### 8.1.3 Google Trends 数据性质与注意事项（官方 FAQ）

- 数据本质：Trends 基于样本化的搜索请求，输出经标准化的相对指数（0–100）。同一主题在不同地域/时间窗口下的 100 并非同一绝对量纲。
- 正常波动：因抽样与防刷机制，多次抓取会有细微差异；低量级查询更易波动与置零。
- 可比性前提：跨时间/地区/设备比较需固定 `timeframe/geo/gprop` 与 `category/topic`，否则尺度不同导致不可比。
- 对比上限：同次对比最多 5 个关键词（或 Topics）。优先使用“主题（Topic）”聚合同义词，必要时用“分类（Category）”限定垂直领域。
- 实务建议：
  - 同一批比较保持完全一致的 `geo/timeframe/category/gprop`；
  - 避免把 `now 7-d` 与 `today 12-m` 放在同一次对比中；跨窗口融合请分开抓取后再做时间对齐与权重融合；
  - 对低量级序列做滚动中位/均值平滑；可多次抓取取中位数以降低抽样噪声；
  - 关注“峰值 100”效应，跨地区比较时建议归一到共同基准期或对照关键词；
  - 结合 `topic` 与 `category` 限定语义，减少歧义与同名实体干扰。
- 官方参考：
  - FAQ about Google Trends data：[support.google.com](https://support.google.com/trends/answer/4365533)
  - Compare search terms（对比与分类）：[support.google.com](https://support.google.com/trends/answer/4359550)

### 8.2 已实现的数据采集组件

**采集器（collectors/）**：
- `BaseCollector`：采集器基类，提供统一接口和速率限制
- `OTACollector`：OTA平台数据采集器，支持CSV和API两种模式

**处理器（processors/）**：
- `DataMatcher`：实体匹配器，支持精确匹配和模糊匹配（相似度阈值0.8）

**数据流**：
```
外部数据源（CSV/API）
    ↓
collectors（采集 + 解析）
    ↓
processors（匹配到内部ID）
    ↓
content_standard表（标准化存储）
    ↓
推荐系统消费
```
