# 测试与验收方案

> 作用：用于指导“内蒙古旅游智能推荐与出行指南系统”的功能测试和最终验收，确保端到端流程可用、核心功能正确、多源数据与推荐算法按预期工作。  
> 约定：后续如有功能调整或新模块上线，应在本文件中补充或更新对应测试用例，不单独新建其他测试文档。

## 1. 测试范围与目标

### 1.1 测试范围

- 后端接口层（Flask + SQLAlchemy，默认 SQLite/可选 MySQL）：
  - 用户注册/登录
  - 资源浏览（景点/酒店/美食）
  - 评分与行为日志
  - 用户画像（偏好设置 + 聚合画像）
  - 推荐接口（热门 + 多源融合 + 相似景点）
  - 行程规划接口
- 前端 Web 应用（Vue）：
  - 注册/登录页面
  - 首页推荐（推荐理由展示）
  - 目的地浏览（评分 + 查看相似景点）
  - 我的偏好
  - 行程规划页面

### 1.2 测试目标

- 验证所有列出的功能均可在“正常路径”下稳定运行；
- 确认多源数据（OTA、社交媒体）已成功接入并影响推荐排序；
- 确保核心推荐与行程规划逻辑在典型场景下结果合理；
- 为最终验收与答辩提供可复现的测试记录。

## 3. 功能测试用例（前端视角）

> 使用前端页面进行人工测试，按用例执行并记录结果。

### 3.1 用户注册与登录

- **用例 3.1-1：正常注册**
  - 前置条件：无同名用户。
  - 步骤：
    1. 在“注册”页面输入用户名 `test_user`，密码 `123456`，邮箱任意。
    2. 点击“注册”。
  - 预期结果：
    - 页面提示“注册成功”或类似文案；
    - 后端 `user` 表中新增记录。

- **用例 3.1-2：重复用户名注册**
  - 步骤：使用已存在用户名再次注册。
  - 预期结果：
    - 前端提示“用户名已存在”或类似错误信息；
    - 数据库中不新增重复用户。

- **用例 3.1-3：登录成功**
  - 步骤：
    1. 在“登录”页面输入正确的用户名和密码；
    2. 点击“登录”。
  - 预期结果：
    - 跳转到首页或提示“登录成功”；
    - 浏览器 LocalStorage 中写入当前用户信息。

- **用例 3.1-4：登录失败**
  - 步骤：使用错误的密码进行登录。
  - 预期结果：
    - 前端显示“用户名或密码错误”；
    - 不写入 LocalStorage。

### 3.2 目的地浏览与评分

- **用例 3.2-1：景点列表浏览**
  - 步骤：
    1. 点击“浏览”；
    2. 类型选择“景点”，城市留空或输入“包头”；
    3. 点击“搜索”。
  - 预期结果：
    - 列表中展示至少 1 条景点卡片（如“希拉穆仁草原”等）。

- **用例 3.2-2：酒店与美食浏览**
  - 步骤：切换类型为“酒店”“美食”，分别搜索。
  - 预期结果：
    - 列表展示对应类型的酒店/美食信息，人均/菜系等字段显示正常。

- **用例 3.2-3：登录后评分成功**
  - 前置条件：已登录。
  - 步骤：
    1. 在景点列表中任意点击“我喜欢（打 5 分）”；
    2. 观察提示。
  - 预期结果：
    - 弹出“感谢你的评分！推荐将逐渐向你的偏好靠拢。”；
    - `rating` 表中新增对应记录；
    - 对同一景点重复评分时，行为符合接口设计（可覆盖或新增，视实现情况）。

- **用例 3.2-4：未登录评分拦截**
  - 前置条件：未登录。
  - 步骤：点击“我喜欢（打 5 分）”。
  - 预期结果：
    - 弹出“请先登录再进行评分”；
    - `rating` 表中无新记录。

### 3.3 我的偏好设置

- **用例 3.3-1：对话式填写偏好并生成行程草案（登录）**
  - 前置条件：已登录。
  - 步骤：
    1. 点击“我的偏好”进入对话页；
    2. 在输入框发送：`想自驾去呼和浩特玩3天，预算3000，亲子`；
    3. 观察左侧对话回复与右侧卡片（Slots/Profile/Itinerary）。
  - 预期结果：
    - 对话区出现助手回复 `reply`；
    - 右侧 `Slots` 卡片展示 `destination/days/budget_level/transport_mode/travel_style/interests` 等字段（未识别项可为 `—`）；
    - 当识别到 `destination` 时，右侧出现 `Itinerary` 草案；
    - 后端响应包含 `profile`（非空）并写入数据库：`user_profile` 表中对应用户记录被创建/更新。

- **用例 3.3-2：对话式生成行程草案（未登录）**
  - 前置条件：未登录。
  - 步骤：
    1. 打开“我的偏好”；
    2. 发送：`去呼伦贝尔玩4天，想看草原和湖泊，预算5000`。
  - 预期结果：
    - 仍可看到 `Slots` 与 `Itinerary` 草案；
    - 页面提示“未登录不会写入画像”；
    - 后端响应 `profile` 为 `null`，数据库 `user_profile` 不发生写入。

- **用例 3.3-3：清空对话并重新开始**
  - 步骤：
    1. 进行若干轮对话；
    2. 点击“清空对话”。
  - 预期结果：
    - 对话恢复到初始问候；
    - 右侧 `Slots` 与 `Itinerary` 置空。

- **用例 3.3-4：输入过长拦截（安全）**
  - 步骤：发送超长文本（长度 > `AGENT_MAX_INPUT_CHARS`，默认 800 字符）。
  - 预期结果：
    - 前端显示错误提示 `输入过长，请简化描述后再试`；
    - 不生成新的行程草案。

- **用例 3.3-5：对话轮次过多拦截（安全）**
  - 步骤：连续对话超过 `AGENT_MAX_TURNS`（默认 20 个用户回合）。
  - 预期结果：
    - 接口返回 `对话轮次过多，请清空对话后继续`；
    - 前端显示错误提示。

### 3.4 首页推荐与推荐理由

- **用例 3.4-1：未登录首页推荐（热门）**
  - 前置条件：未登录。
  - 步骤：进入首页。
  - 预期结果：
    - “推荐景点”区域显示一组景点；

    - 若多源数据已经写入，则部分卡片显示“推荐理由”（来自 OTA 与社交媒体）。

- **用例 3.4-2：登录 + 有偏好 + 有评分后的推荐**
  - 前置条件：
    - 已登录；
    - 已在“我的偏好”设置若干偏好景点类型；
    - 在“浏览”中对这些类型的景点评分过。
  - 步骤：回到首页，观察“推荐景点”。
  - 预期结果：
    - 偏好类型的景点排序靠前；
    - “推荐理由”中出现“符合你的偏好类型：XXX”等文案；

### 3.5 相似景点推荐

- **用例 3.5-1：查看相似景点**
  - 前置条件：已运行 `aggregate_scenic_cf.py`，且有景点评分数据。

## 4. 接口层测试（后端 API）

> 建议使用 Postman / curl 对关键接口做简单验证，以防前端问题掩盖接口问题。

### 4.1 健康检查

- `GET /api/health` → 预期：`{"status": "ok"}`。

### 4.2 认证与用户数据接口

- `POST /api/auth/register`：
  - 请求体示例：

    ```json
    {
      "username": "test_user",
      "password": "123456",
      "email": "test_user@example.com",
      "phone": "13800000000"
    }
    ```

  - 预期：返回 201，响应体包含 `id/username/email/phone/created_at`，且不包含密码/哈希等敏感字段。
  - 参数校验与错误码：
    - `username/password` 缺失 → 400 且包含 `{"error": "username and password are required"}`；
    - 用户名长度 < 3 或密码长度 < 6 → 400 且包含 `{"error": "username must be >=3 chars and password >=6 chars"}`；
    - `username` 已存在 → 400 且包含 `{"error": "username already exists"}`；
    - `email` 已存在 → 400 且包含 `{"error": "email already exists"}`。

- `POST /api/auth/login`：
  - 请求体示例：

    ```json
    {"username": "test_user", "password": "123456"}
    ```

  - 预期：返回 200，响应体包含 `message = "login_success"` 及 `user` 对象（`id/username/email/phone`）。
  - 参数校验与错误码：
    - 缺失 `username/password` → 400 且包含 `{"error": "username and password are required"}`；
    - 账号或密码错误 → 400 且包含 `{"error": "invalid username or password"}`。

- `POST /api/ratings`：
  - 请求体示例：

    ```json
    {
      "user_id": 1,
      "target_type": "scenic_spot",
      "target_id": 3,
      "score": 5,
      "comment": "很不错的草原体验"
    }
    ```

  - 预期：返回 201，响应体包含 `rating` 与 `aggregate`（`rating_avg/rating_count`），且目标对象（景点/酒店/美食）的 `rating_avg/rating_count` 被更新。
  - 参数校验与错误码：
    - 必填项缺失 → 400 且包含 `{"error": "user_id, target_type, target_id and score are required"}`；
    - `user_id/target_id/score` 非整数 → 400 且包含 `{"error": "user_id, target_id and score must be integers"}`；
    - `score` 不在 1~5 → 400 且包含 `{"error": "score must be between 1 and 5"}`；
    - `target_type` 不在 `scenic_spot/hotel/food_place` → 400 且包含 `{"error": "unsupported target_type"}`；
    - `user_id` 不存在 → 400 且包含 `{"error": "user not found"}`。

- `POST /api/behaviors`：
  - 预期：返回 201，写入行为日志（未登录 `user_id` 可为空）。
  - 参数校验与错误码：
    - 缺失 `target_type/target_id/behavior_type` → 400 且包含 `{"error": "target_type, target_id and behavior_type are required"}`；
    - `target_id` 非整数 → 400 且包含 `{"error": "target_id must be an integer"}`。

- `GET /api/user/profile/{user_id}`：
  - 预期：
    - 若用户画像不存在，返回 `{"profile": null}`；
    - 若存在，返回 `{"profile": {...}}`（包含 `prefer_scenic_types/prefer_food_types/travel_style/budget_level` 等字段）。

- `POST /api/user/profile`：
  - 请求体示例：

    ```json
    {
      "user_id": 1,
      "travel_style": "relax",
      "budget_level": 2,
      "prefer_scenic_types": ["草原", "森林"],
      "prefer_food_types": "烧烤,牛肉"
    }
    ```

  - 预期：返回 200，响应体包含 `profile` 对象，且可被 `GET /api/user/profile/{user_id}` 查询到。
  - 参数校验与错误码：
    - `user_id` 非法 → 400 且包含 `{"error": "valid user_id is required"}`；
    - 用户不存在 → 400 且包含 `{"error": "user not found"}`；
    - `budget_level` 非整数 → 400 且包含 `{"error": "budget_level must be an integer"}`；
    - `budget_level` 不在 1/2/3 → 400 且包含 `{"error": "budget_level must be 1, 2, or 3"}`。

### 4.3 核心资源接口

- `GET /api/scenic-spots`：
  - 无参数：应返回分页景点列表；
  - 带 `city` 参数：只返回指定城市景点。
- `GET /api/hotels`、`GET /api/foods`：
  - 测试 `city`、价格区间、菜系等参数是否生效。

### 4.4 推荐接口

- `GET /api/recommend/scenic-spots?limit=10`：
  - 无 `user_id`；检查返回 items 结构和排序基本合理；
  - 部分条目应有 `reasons` 字段。
- `GET /api/recommend/scenic-spots?user_id={uid}&limit=10`：
  - 用户画像存在时，检查偏好类型是否影响排序；
- `GET /api/recommend/scenic-spots/{spot_id}/similar?limit=5`：
  - 检查返回的相似景点列表与评分数据是否合理。

### 4.5 行程规划接口

- `POST /api/route/plan`：
  - 请求体示例：

    ```json
    {
      "spot_ids": [1, 2, 3],
      "start_spot_id": 1,
      "optimize": "greedy"
    }
    ```

  - 预期：响应体包含 `route`（按游览顺序的景点数组）与 `meta`（元信息）字段；其中 `meta.total_distance_km` 为总里程（km，可能来自 AMap 或 Haversine 近似）。
  - 参数校验与错误码：
    - `spot_ids` 非数组或长度 < 2 → 预期返回 400 且包含 `{"error": "spot_ids must be a list with at least two ids"}`；
    - `spot_ids` 含非整数 → 预期返回 400 且包含 `{"error": "spot_ids must contain integers"}`；
    - `start_spot_id` 非整数 → 预期返回 400 且包含 `{"error": "start_spot_id must be an integer"}`；
    - 至少两条 ID 在库中不存在 → 预期返回 400 且包含 `{"error": "at least two scenic spots must exist"}`；
    - 部分 ID 不存在但仍能规划（存在 >=2 个）→ 预期 `meta.missing_ids` 返回缺失 ID 列表。

### 4.6 路线方案与运行指标（新增）

- `POST /api/route/options`：
  - 前置：如需真实 AMap 结果，配置环境变量 `AMAP_WEB_KEY`（可选 `AMAP_API_VERSION=v3|v5`）。
  - 驾车示例：

    ```json
    {
      "origin": {"lng": 116.4343, "lat": 39.9091},
      "destination": {"lng": 116.4344, "lat": 39.9082},
      "mode": "drive"
    }
    ```

  - 公交换乘（同城，v5 示例，可选请求覆盖）：

    ```json
    {
      "origin": {"lng": 116.4815, "lat": 39.9905},
      "destination": {"lng": 116.4651, "lat": 39.9995},
      "mode": "transit",
      "api_version": "v5",
      "city1": "010",
      "city2": "010",
      "transit_strategy": "0"
    }
    ```

  - 预期：返回 `{"options": [...]}`，每项含 `label/duration_min/distance_km/legs`。无 Key 或外呼失败时，将回退到 `v3/distance` 或 Haversine 占位方案。

- `GET /api/route/metrics`：
  - 用途：检查调用成功/失败、回退、缓存命中/未命中、节流次数、推断 citycode 等指标。
  - 预期：响应包含 `uptime_sec`、`metrics` 和 `summary`；其中 `summary` 应包含：
    - `amap_success_total`、`amap_fail_total`、`amap_success_rate`
    - `route_options_total`、`source_breakdown`
    - `fallback_amap_distance`、`fallback_haversine`
    - `amap_cache_hit`、`amap_cache_miss`、`amap_rate_limited_skips`
    - `infer_citycode_used`

### 4.7 偏好对话 Agent（新增）

- `POST /api/agent/chat`：
  - 用途：自然语言偏好对话，抽取 `slots`，可选更新画像，并返回行程草案预览。
  - 登录示例（含 messages）：

    ```json
    {
      "user_id": 1,
      "text": "想自驾去呼和浩特玩3天，预算3000，亲子",
      "messages": [
        {"role": "assistant", "content": "你好！请描述你的出行偏好..."},
        {"role": "user", "content": "想自驾去呼和浩特玩3天，预算3000，亲子"}
      ]
    }
    ```

  - 预期：
    - 返回体包含 `reply/slots/actions/profile/itinerary` 字段；
    - `slots.destination` 为城市名时，`itinerary.days` 应为数组；
    - 传 `user_id` 且识别到 `budget_level/travel_style/interests` 时，`profile` 非空。

  - 未登录示例：

    ```json
    {"text": "去呼伦贝尔玩4天，想看草原，预算5000"}
    ```

  - 预期：
    - 仍返回 `slots/itinerary`（当存在 destination）；
    - `profile` 为 `null`。

  - 安全用例：
    - 输入长度 > `AGENT_MAX_INPUT_CHARS`（默认 800）→ 预期返回 400 且 `{"error": "输入过长，请简化描述后再试"}`；
    - `messages` 中用户回合数 > `AGENT_MAX_TURNS`（默认 20）→ 预期返回 400 且 `{"error": "对话轮次过多，请清空对话后继续"}`。

- 补充用例（v5 驾车 + waypoints 多途经点）：

  ```json
  {
    "origin": {"lng": 116.4343, "lat": 39.9091},
    "destination": {"lng": 116.4344, "lat": 39.9082},
    "mode": "drive",
    "api_version": "v5",
    "waypoints": [
      {"lng": 116.401, "lat": 39.901},
      [116.402, 39.902],
      "116.403,39.903"
    ]
  }
  ```

  - 预期：
    - 返回 `options`（正常外呼时）；
    - 服务端会将 `waypoints` 序列化为分号分隔的串（最多 16 个途经点）；
    - 当 `waypoints` 超过 16 个时，返回 400 且提示 `waypoints exceed 16`。

- 补充用例（v5 公交 + ad1/ad2 + AlternativeRoute/multiexport/nightflag）：

  ```json
  {
    "origin": {"lng": 116.4815, "lat": 39.9905},
    "destination": {"lng": 116.4651, "lat": 39.9995},
    "mode": "transit",
    "api_version": "v5",
    "city1": "010",
    "city2": "010",
    "ad1": "110105",
    "ad2": "110108",
    "transit_strategy": "0",
    "alternative_route": 3,
    "multiexport": 1,
    "nightflag": 1
  }
  ```

  - 预期：
    - 返回 `options`（正常外呼时）；
    - `city1/city2` 必填（citycode），`ad1/ad2` 为可选（adcode）；
    - `AlternativeRoute` 取值 1–10，`multiexport/nightflag` 取值 0/1；非法取值会被忽略（不出现在请求中）。

### 4.8 行程草案接口（Itinerary）

- `POST /api/itineraries/suggest`：
  - 请求体示例：

    ```json
    {
      "city": "包头",
      "user_id": 1,
      "days": 3,
      "candidates": {
        "scenic_spot": [1, 2, 3],
        "food_place": [10, 11]
      }
    }
    ```

  - 预期：响应包含 `itinerary.days` 数组；每个 day 包含 `day_index/items`；item 包含 `type/id/start_time/end_time`。
  - 参数校验与错误码：
    - `user_id` 非整数 → 400 且包含 `{"error": "user_id 必须为整数"}`；
    - `days` 非整数 → 400 且包含 `{"error": "days 必须为整数"}`；
    - `candidates` 非对象 → 400 且包含 `{"error": "candidates 必须为对象"}`；
    - 当前无可用内容 → 400 且包含 `{"error": "当前暂无可用内容，无法生成行程"}`。

## 5. 简单性能、稳定性与安全性测试

> 本项目不做严格性能压测，仅做轻量级检查。

- 在本机环境：
  - 连续刷新首页和浏览页 20~50 次，确认无明显卡顿和接口错误；
  - 行程规划接口对 2~10 个景点的响应时间应在 1s 级别（不含外部地图 API）。
- 若配置了高德 API：
  - 观察多次调用后是否有超时或频率限制问题（如有需在文档说明）。

- 安全性快速检查：
  - 注册/登录接口返回体不应包含密码/哈希等敏感字段；
  - 关键接口对明显非法输入应返回 400（而非 500），错误信息清晰；
  - `/api/agent/chat` 超长输入与轮次限制应生效（对应 4.7 安全用例）。

### 6.2 非功能验收矩阵（性能/稳定性/安全）

| 类别 | 检查项 | 方法 | 通过标准 |
|---|---|---|---|
| 性能 | 列表接口响应 | 连续请求 `GET /api/scenic-spots` 等 30 次 | 本机环境下整体响应无明显卡顿；单次响应通常在 1s 内 |

系统可被视为“通过验收”的条件示例：

1. 所有 3.x 章节的功能测试用例通过率 ≥ 95%，无阻塞缺陷；
2. 核心接口（认证、浏览、评分、推荐、相似景点、行程规划）均可通过 Postman 正常访问；

4. 行程规划结果在典型场景下无明显逻辑错误；
5. 系统在持续操作 30 分钟内无崩溃或严重错误日志。
6. 对话式“我的偏好”可用：
   - 能稳定返回 `slots` 并在提供目的地时生成行程草案；
   - 登录状态下能按需更新 `user_profile`；未登录不写入；
   - 输入长度限制与对话轮次限制生效。

7. 安全性基本检查通过：
   - 注册/登录响应不泄露密码/哈希等敏感字段；
   - 关键接口对非法输入均返回 400（非 500）。
