## 1. 推荐算法总体设计概览

### 1.1 当前已实现的算法组件

1. **热门推荐（Baseline 1）**  
   - 在线服务中：
     - 依托脚本 `aggregate_content.py` 聚合得到的 `content_standard.popularity_score` 进行排序；
     - 该分数由受欢迎度回归模型 `popularity_model.pkl` 预测，当前模型通过脚本 `train_popularity_model.py` 基于“多源特征”（内部评分/OTA/社交）训练，并使用“多源组合标签”（`base_popularity + ota_rating + social_heat + 2 * social_sentiment`）作为监督信号；
     - 训练偏好：优先使用 LightGBM（启用单调约束 monotone constraints），环境不可用时回退 RandomForest；
     - 特征工程一致性：训练端可选 `--log-transform` 对计数特征（`rating_count/ota_review_count/social_interactions`）做 log1p 变换，并将开关写入 `models/popularity_model.meta.json`；聚合端将读取该元信息并一致应用；
     - 在线回退：如某实体缺少 `internal_rating` 聚合结果，则使用 IMDb 风格的 Bayesian WR 即时计算内部基线，公式 `WR = (v/(v+m))*R + (m/(v+m))*C`（C 为全局评分均值，m 为评分数分位阈值）。
     - 若模型文件不存在或加载失败，`aggregate_content.py` 将直接报错并中止聚合，提示先完成模型训练，不再自动回退到规则公式。
   - 离线评估基线：
     - 直接基于内部评分与评分次数排序：
       - 按 `rating_avg` 降序；
       - 再按 `rating_count` 降序；
       - 再按 `id` 降序（稳定排序）。

2. **画像驱动的类型偏好推荐（Baseline 2）**  
   - 通过 `UserProfile.prefer_scenic_types` 与 `prefer_food_types` 等字段记录显式偏好；
   - 推荐时：
     - 先取一批候选（按热门排序）；
     - 将类型属于偏好集合的内容优先排序，再在集合内按评分与人气排序。

3. **多源数据融合推荐（Model 1）**  
   - 在景点推荐接口 `/api/recommend/scenic-spots` 中融合三类数据：
     - 内部评分聚合的 `popularity_score`（`source_type = "internal_rating"`）；
     - OTA 真实样本数据（人工整理）对应的外部评分与评论数（`source_type = "ota_stats"`，兼容历史 `ota`；读取时兼容 `external_rating/rating` 与 `review_count/external_review_count`）；
     - 社交热度/舆情数据的 `interaction_sum` 与 `sentiment_avg`（`source_type = "social_media"`）。
   - 通过 `_compute_multi_source_score` 计算综合得分：
     - 内部受欢迎度（评分 + 人气）；
     - 外部评分（OTA）；
     - 社交媒体热度与情感（缩放后加入）。
   - 有用户画像时，在类型偏好基础上再按综合得分排序。

4. **基于 item-based 协同过滤的相似景点推荐（Model 2）**  
   - 离线脚本 `aggregate_scenic_cf.py`：
     - 基于 `rating` 中用户-景点评分矩阵，计算景点间的余弦相似度；
     - 为每个景点生成相似邻居列表，并写入 `content_standard`：
       - `entity_type = "scenic_spot"`
       - `source_type = "item_cf_similarity"`
       - `summary` JSON 中含有 `neighbors: [{"id": scenic_id, "sim": similarity}, ...]`.  
   - 在线接口 `/api/recommend/scenic-spots/{spot_id}/similar`：
     - 按相似度 + 评分 + 人气排序返回相似景点；
     - 前端在“目的地浏览”页中通过“查看相似景点”按钮展示结果。

5. **离线评估脚本（evaluate_recommendation.py）**  
   - 用于在当前数据库数据基础上，对不同推荐策略进行离线对比；
   - 指标重点关注：Top-K 命中情况（hit / precision），后续可扩展 recall 与 MAP；
   - 作为论文和答辩中“算法效果”章节的实验支撑。

6. **矩阵分解推荐（Model 3，Biased MF）**  
   - 离线脚本 `evaluate_recommendation.py` 中：
     - 使用带全局均值、用户偏置、物品偏置的 Biased MF 训练用户与景点隐向量；
     - 对每个用户在“尚未评分”的景点集合上做 Top-K 推荐，与 Popular / Profile / CF 做精度对比。
   - 在线接口 `/api/recommend/scenic-spots?strategy=mf`：
     - 在请求时基于当前评分表训练一个轻量级 MF 模型；
     - 使用与离线一致的预测公式 `global_mean + b_u + b_i + <p_u, q_i>` 计算候选景点得分并排序。

7. **Hybrid 混合打分推荐（Model 4，Hybrid）**  
   - 在线接口 `/api/recommend/scenic-spots?strategy=hybrid`：
     - 对同一批候选景点综合三路得分：
       - 多源基线分（内部评分 + OTA + 社交媒体）；
       - 基于 item-based CF 的相似度加权得分；
       - Biased MF 预测评分；
     - 在候选集上分别对三路得分做 min-max 归一化后，以 `w_base = 0.2, w_cf = 0.4, w_mf = 0.4` 加权求和得到总分；
     - 若某一路在当前候选集无有效信号，则自动对剩余通道重归一权重.
   - 离线脚本 `evaluate_recommendation.py` 中使用相同配方，对 Hybrid 的 Precision@K 进行评估，对比 Popular / Profile / CF / MF.

### 1.2 后续可扩展方向（规划）

- 混合推荐：将热门、画像、协同过滤、多源特征等按权重融合；
- 排序模型：使用逻辑回归 / XGBoost 对候选列表进行二次排序；
- 时间维度：引入季节性与节假日因素，做“时间感知”的推荐.

## 2. 离线评估脚本设计（evaluate_recommendation.py）

### 2.1 评估目标

- 在现有评分数据基础上，对不同推荐策略进行离线对比；
- 指标重点关注：Top-K 命中情况（hit / precision），后续可扩展 recall 与 MAP；
- 作为论文和答辩中“算法效果”章节的实验支撑.

### 2.2 数据与划分方式

1. 数据来源：
   - `rating` 表中 `target_type = 'scenic_spot'` 的评分记录；
2. 用户筛选：
   - 仅保留评分数量 ≥ 3 的用户参与评估；
3. 时间划分策略：
   - 对每个用户，按 `created_at` 时间升序排序评分记录；
   - 将最后一条评分作为“测试集”；
   - 之前的评分作为“历史行为（训练集）”。

### 2.3 推荐策略定义

1. **热门推荐（Popular）**  
   - 全局 SQL 聚合：
     - 计算每个景点的平均评分与评分次数；
     - 按 `avg_score DESC, cnt DESC` 排序；
     - 作为与用户无关的 Top-N 列表.

2. **热门推荐-WR（Popular-WR）**  
   - 使用 IMDb 风格 Bayesian WR 进行排序：
     - `WR = (v/(v+m))*R + (m/(v+m))*C`；
     - 其中 C 为全局评分均值，m 取评分数的 60% 分位（低于 1 则置为 1）；
   - 作为与用户无关、对小样本更鲁棒的热门基线.

3. **画像推荐（Profile-Based）**  
   - 读取用户画像表 `user_profile`：
     - `prefer_scenic_types` 中记录的偏好类型集合；
   - 在一批候选景点中（按评分+人气初步排序）：
     - 将 `category` 落在偏好集合中的景点排在前面；
     - 集合内部按评分、人气排序.

4. **协同过滤推荐（CF-Based）**  
   - 读取 `content_standard` 中 `source_type = 'item_cf_similarity'` 的邻居列表；
   - 对用户历史评分过的景点的邻居做“票数累加”；
   - 排除用户已评分项目后，选出 Top-N 作为 CF 推荐结果.

5. **矩阵分解推荐（MF-Based，Biased MF）**  
   - 在 `evaluate_recommendation.py` 中：
     - 使用 2.2 小节的时间划分策略构建训练集与测试集；
     - 以 `(user_id, scenic_id, score)` 为三元组训练带偏置的矩阵分解模型（Biased MF）；
     - 对每个用户，在“尚未评分”的景点集合上按预测评分排序，取 Top-K 作为 MF 推荐结果.

6. **Hybrid 混合推荐（Hybrid, Popular-WR + CF + MF）**  
   - 先基于训练集统计每个景点的“热门基线分”（已改为 Bayesian WR）：
     - `WR = (v/(v+m))*R + (m/(v+m))*C`；
   - 使用 `item_cf_similarity` 中的相似度，对用户历史项目的邻居做相似度加权累加，得到 CF 通道得分；
   - 使用同一套 Biased MF 模型计算候选景点的预测评分，作为 MF 通道得分；
   - 在候选集合上分别对三路分数做 min-max 归一化，再以 `w_base = 0.2, w_cf = 0.4, w_mf = 0.4` 加权求和，得到最终排序结果.

> 注：
> - 纯 CF 策略仍使用“邻居出现次数”作为票数；
> - Hybrid 策略中则基于 `item_cf_similarity` 中的 `sim` 字段做相似度加权求和，并与热门基线分、MF 预测分一起做加权融合.

### 2.4 指标定义

- **Precision@K / NDCG@K**（当前版本已支持）：
  - Precision@K：前 K 个推荐中的命中比例；
  - NDCG@K：考虑命中位置的折损累计增益，使用二值相关性版本；
  - 评估脚本统一生成 Top-10 列表，并同时统计 @5 与 @10 的两组指标.

> 后续可扩展：
> - Recall@K：命中数 / 测试集中项目总数；
> - MAP 等排序质量指标.

### 3.3 实验结果示例表格

> 以下为当前基于**合成评分数据**的一次评估结果（Precision@5）。该结果用于验证“策略实现 + 评估链路”正确性；正式结论应以**真实评分/行为日志**为主。
> 合成评分可通过脚本 `generate_synthetic_ratings.py` 生成，真实评分建议通过 `import_ratings_from_csv.py` 导入（字段可追溯、可复现）。

| 模型                        | Precision@5 | Precision@10 | 备注                                                   |
|-----------------------------|-------------|--------------|--------------------------------------------------------|
| Popular（热门推荐）         | 0.1824      | 0.1000       | 基于评分次数与平均分的全局热门                        |
| Popular‑WR（Bayesian WR）   | 0.1824      | 0.1000       | 使用 WR 作为更鲁棒的热门基线                          |
| ML‑Popular（多源模型热门）  | 0.1765      | 0.1000       | 基于 `content_standard.popularity_score`（多源模型预测）|
| Profile-Based（画像）       | 0.1647      | 0.1000       | 简单类型偏好重排，效果略低于热门                      |
| CF-Based（协同过滤）       | 0.2000      | 0.1000       | 基于 item-based CF，相比热门略有提升                  |
| MF（矩阵分解）             | 0.2000      | 0.1000       | 基于潜在因子模型的推荐，性能与 CF 持平略优于热门      |
| Hybrid（多源融合推荐）     | 0.2000      | 0.1000       | 评估脚本与在线一致，Hybrid 的基线已改为 WR            |

### 3.4 结果分析要点（撰写时参考）

**当前基于合成评分数据的简要结论：**

- 在 34 名用户、每人 20~50 条景点评分的合成数据上，CF-Based 与 MF 的 Precision@5 均为 0.2000，略高于 Popular 的 0.1824 和 Profile-Based 的 0.1647，说明在该合成数据的设置下，基于协同过滤和潜在因子模型的推荐，能够从评分结构中获得一定精度提升，但提升幅度相对温和.  
- 画像推荐（Profile-Based）略低于热门推荐，原因在于当前画像构造与合成评分生成规则较为简单，且用户偏好类型与热门景点高度重合，导致“按类型重排”并未显著改变 Top-5 命中率；在接入真实画像或更复杂的偏好建模后，预计画像方法的相对优势会更明显.  
- 需要强调的是，本轮实验基于合成评分数据，主要目的在于验证评估链路与算法实现正确性，并提供一个可复现的对比示例；未来在接入真实评分与行为数据后，应重新运行评估并以真实数据结果作为论文和答辩中的核心结论。

**撰写正式报告/论文时可展开的分析要点：**

- 热门推荐的优点与局限：
  - 优点：实现简单，冷启动友好；
  - 局限：个性化不足，容易集中在头部景点。
- 画像推荐的提升点：
  - 对特定类型偏好用户的点击/满意度提升情况；
- 协同过滤的优势：
  - 对“长尾景点”的挖掘能力；
  - 对于评分较多用户，推荐准确性提升；
- 多源融合策略的价值：
  - 结合 OTA 与社交媒体数据后，对“真实热门且口碑好”景点的识别效果；
  - 是否减少单一数据源偏差带来的影响。

### 3.5 基于公开数据集的补充离线实验（示例：Yelp Open Dataset）

> 说明：本节用于记录基于公开推荐数据集（如 Yelp Open Dataset）开展的**补充离线实验**。这些实验不接入项目自身数据库，仅在独立脚本/notebook 中完成，用于验证热门推荐等算法在大规模真实数据上的表现，为本系统后续算法设计提供参考。

- 数据集示例：Yelp Open Dataset
  - 字段：`user_id, business_id, stars, date, ...`；
  - 只使用 `user_id`、`business_id`、`stars`、`date` 四个字段构建评分记录；
  - 数据文件 `review.json` 建议放在项目根目录下：`E:\旅游\data\public\yelp\review.json`（逐行 JSON）。

- 实验脚本：`app/pipelines/yelp_offline_experiment.py`
  - 功能：
    - 从 `review.json` 中加载前 N 条评分记录（可通过 `--max-reviews` 控制）；
    - 按“每个用户最后一次评分为测试集”的策略构建 leave-one-out 训练/测试划分；
    - 基于训练集统计全局热门项目（平均评分 + 评分次数），计算 Popular@K 的命中率（HitRate@K）。
  - 运行方式（在 `E:\旅游\backend` 下）：

```powershell
python -m app.pipelines.yelp_offline_experiment \
  --data-dir "E:\旅游\data\public\yelp" \
  --max-reviews 200000 \
  --topk 10
```

## 4. 可解释性与线上可观测性（需求对齐）

### 4.1 推荐解释性（`reasons` 字段）

- 在线接口：`GET /api/recommend/scenic-spots` 返回的景点条目可能包含 `reasons`（字符串数组）。
- 解释来源（与当前实现对齐）：
  - 内部评分：当 `rating_avg/rating_count` 达到阈值时，出现 `本站评分较高`；
  - OTA 外部评分：当 `ota_rating/ota_review_count` 达到阈值时，出现 `OTA 平台评分优秀/较好`；
  - 社交热度：当 `social_interactions` 达到阈值时，出现 `社交媒体热度很高/讨论度较高`；
  - 情感：当 `social_sentiment` 达到阈值时，出现 `网友整体评价非常好/较好`；
  - 画像：当用户偏好类型命中时，出现 `符合你的偏好类型：{category}`。

> 说明：该解释为“工程可落地的可解释性”，用于支撑系统演示与验收；后续可扩展为更精细的贡献度解释（如分数分解、特征贡献 Top-N）。

### 4.2 线上可观测性（行为日志 → 分析闭环）

- 线上行为日志：后端提供 `POST /api/behaviors` 写入 `user_behavior_log`。
  - 推荐相关建议埋点：
    - 曝光（expose/view）：当推荐列表渲染完成后记录；
    - 点击（click）：用户点击某条推荐卡片时记录；
    - 行为值（`behavior_value`）：可用于记录停留时长等数值。
- 离线分析脚本：`app/pipelines/analyze_behavior.py`
  - 用途：统计最近 N 天的点击 TopN、行为类型分布、目标类型分布、每日趋势等。
  - 输出可直接作为“线上可观测性”的答辩证据（推荐后是否被用户点击、是否产生转化）。

### 4.3 需求导向的评估指标建议（落地口径）

- 离线 Top‑K 排序质量（已实现）：Precision@K、NDCG@K（脚本 `evaluate_recommendation.py`）。
- 覆盖与数据新鲜度（建议作为必报指标）：
  - Coverage：`ContentStandard` 中 `ota_stats/social_media` 的覆盖率（脚本 `analyze_feature_coverage.py`）；
  - Freshness：按 `content_standard.last_update` 统计外部特征是否过期（建议在覆盖率脚本中补充时间分布统计）。
- 在线指标（可从行为日志计算）：
  - CTR@K：推荐曝光→点击转化率（整体 + 分城市/分类型）；
  - Rating conversion：点击后是否产生评分（可用 `rating` 表或行为日志关联）。

## 5. 数据工程与算法深化规划

> 本节用于规划后续“数据工程 / 算法深化阶段”的实现路线。随着工作推进，应在本节中持续更新进展与调整，不再另建新的规划文档。
### 5.1 多源数据工程规划

目标：从当前示例 CSV 级别的 OTA / 社交媒体数据，逐步演进为可扩展的多源数据工程框架，与项目书中提出的“淘宝、抖音、微博、小红书、交通大数据、官方统计”等保持一致。

- **数据源分类与落地表设计**：
  - 社交媒体与 UGC：微博、小红书、短视频平台等 → 舆情热度、情感得分、话题标签；
  - OTA / 电商：携程、去哪儿、美团、淘宝等 → 评分、评论数、销量、人均价格等；
  - 交通与客流：12306、道路车流量、景区客流 → 节假日拥挤度、出行便利度；
  - 官方统计与文献：文旅厅数据、统计局指标、CNKI 文献 → 城市/区域级宏观指标。
- **标准化与映射规则**：
  - 统一通过 `entity_type + entity_id`（或城市/区域 ID）映射到内部实体；
  - 以 `content_standard` 为核心标准化表，按 `source_type` 区分不同来源；
  - 为每类源定义字段规范（字段名、含义、单位、时间窗口）。
- **pipeline 规划**：
  - 保持当前 OTA pipeline（`backend/data/ota_sample.csv` + `OTACollector` + `load_ota_data.py`）与社交媒体骨架（`social_media_fetcher.py`）的结构；
  - 按数据源逐步扩展新的 fetcher / aggregator 脚本，输出至统一的内部 schema；
  - 所有采集与落地脚本在《数据采集与处理方案.md》中保持同步描述。

> 后续当某一类真实数据源接入时，应在本节 5.1 中补充“数据字段 + pipeline 名称 + 落地表/字段”的说明。

### 5.2 特征工程与画像深化规划

目标：将目前较为分散的用户/景点信息与多源指标，系统化为可用于机器学习模型的特征向量或特征 JSON，支撑矩阵分解、排序模型、深度学习等多种算法。

- **用户特征（User Features）**：
  - 人口属性：性别、年龄段、地域（来自 `user`）；
  - 行为统计：评分次数、活跃天数、最近一次评分时间、浏览/评分时长等；
  - 内容偏好：景点类型、美食类型、出行风格、预算等级（来自 `user_profile`）；
  - 多源标签：社交媒体活跃度、特定话题关注度（预留字段）。
- **项目（景点/酒店/美食）特征（Item Features）**：
  - 基本属性：类别、城市、价位区间、标签、级别等；
  - 内部指标：评分均值、评分数、popularity_score；
  - 外部指标：OTA 评分/评论数、社交媒体热度/情感分、交通/客流与官方指数；
  - 文本特征：名称、简介、标签的分词与主题（可后续引入）。
- **特征聚合 pipeline 规划**：
  - 新增一个特征聚合脚本（如 `aggregate_feature_vectors.py`，命名可在实现时最终确定）：
    - 从各业务表和 `content_standard` 中读取多源指标；
    - 生成 `UserProfile.feature_vector`（或 `feature_json`）；
    - 为景点/酒店/美食生成统一的 `item_feature_vector`，可存放在 `content_standard` 或新特征表中；
  - 在本文件与《接口与模块设计.md》中记录特征结构与字段含义。

> 后续在实现特征聚合 pipeline 时，应在本节 5.2 中补充“具体字段列表 + 存储位置 + 更新频率”的说明，并在评估脚本中注明使用了哪些特征。

### 5.3 模型扩展与评估规划

目标：在当前热门/画像/CF + 多源融合的基础上，引入至少一类“模型化”的推荐方法（如矩阵分解、排序模型），并与既有策略进行系统对比。

- **矩阵分解（MF / Latent Factor）规划**：
  - 在 `app/pipelines/` 下新增 MF 脚本（如 `aggregate_scenic_mf.py`）：
    - 读取 `rating` 中景点评分记录；
    - 采用时间划分策略构建 train/test（复用 2.2 节划分方式）；
    - 使用 SGD 训练简化版用户/景点隐向量；
    - 为每个用户生成 Top-N 推荐结果写入 `content_standard`（`source_type = "mf_recommendation"`）；
  - 在 `evaluate_recommendation.py` 中增加 MF 策略的 Precision@K，对比 Popular / Profile-Based / CF。
- **排序模型（Learning-to-Rank）规划**：
  - 利用 5.2 中的用户/项目特征，构建简单的二分类/排序任务：预测用户对项目的偏好概率；
  - 初期可使用逻辑回归或梯度提升树（如 XGBoost）进行实验；
  - 在评估脚本中增加对应策略，并记录实验结果在第 3 章表格中。
- **时间与场景扩展**：
  - 引入时间衰减权重与节假日标记，构建时间感知推荐；
  - 将交通与客流数据纳入推荐权重（例如对过于拥挤的景点适度降权）。

> 随着 MF 或排序模型实际落地，应在本节 5.3 中记录：脚本名称、主要超参数、训练数据量、评估指标结果与简要分析，并在第 3 章表格中补充对应行。
