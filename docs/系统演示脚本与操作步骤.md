# 系统演示脚本与操作步骤

> 作用：用于答辩、课堂展示或自测时，按步骤快速跑通“内蒙古旅游智能推荐与出行指南系统”的完整流程。  
> 约定：后续如有功能调整或流程变化，应在**本文件上直接更新**，而不是另建新文档。

---

## 1. 环境与数据准备

### 1.1 必要环境

- 操作系统：Windows
- 已安装：
  - Python 3.x + 虚拟环境 `venv`
  - Node.js + npm
  - SQLite（默认，`dev.db`）
  - MySQL 5.7/8.x（可选，需设置 `DATABASE_URL`）

### 1.2 确认数据库与表

**默认 SQLite（本地开发）**：

1. 确认 `E:\旅游\dev.db` 存在（仓库已内置）。
2. 若 `dev.db` 缺失或需重建，可依次执行：

   ```powershell
   PS E:\旅游\backend> ..\venv\Scripts\activate
   (venv) PS E:\旅游\backend> python .\scripts\seed_dev.py
   (venv) PS E:\旅游\backend> python .\scripts\refresh_dev_data.py
   ```

3. 如需快速检查表/数据，可执行：

   ```powershell
   PS E:\旅游> python -m sqlite3 E:\旅游\dev.db
   sqlite> .tables
   sqlite> SELECT COUNT(*) FROM scenic_spot;
   ```

**MySQL（可选/生产）**：

1. 设置 `DATABASE_URL` 指向 MySQL（如 `mysql+pymysql://.../tourism_imu`）。
2. 启动 MySQL 服务。
3. 使用客户端或命令行连接 MySQL，确认存在数据库 `tourism_imu`：

   ```sql
   SHOW DATABASES;
   USE tourism_imu;
   SHOW TABLES;
   ```

4. 若数据库或表缺失，可使用项目提供脚本重建（仅在需要时）：

   ```powershell
   PS E:\旅游> mysql -u root -p tourism_imu < .\docs\数据库建表.sql
   ```

### 1.3 导入示例基础数据

- **SQLite**：仓库内置 `dev.db` 已包含景点/酒店/美食示例数据。若使用全新数据库，可先运行 `seed_dev.py` 生成示例实体，再按第 2 节执行聚合脚本。
- **MySQL**：若不确定是否已有景点/酒店/美食示例数据，可在 MySQL 中检查：

  ```sql
  USE tourism_imu;
  SELECT COUNT(*) FROM scenic_spot;
  ```

  如数量很少或为 0，可执行：

  ```powershell
  PS E:\旅游> mysql -u root -p tourism_imu < .\data\sample_data.sql
  ```

## 2. 运行离线聚合与多源数据脚本

> 目的：构建推荐所需的内部热门度、用户画像、协同过滤相似度以及 OTA / 社交媒体示例数据。

在 `E:\旅游\backend` 目录下（确保已激活虚拟环境 `venv`）：

```powershell
PS E:\旅游\backend> ..\venv\Scripts\activate
```

提示（SQLite 快速路径）：如需快速生成评分/CF/热门度，可先执行：

```powershell
(venv) PS E:\旅游\backend> python .\scripts\refresh_dev_data.py
```

再按需补充 OTA / 社交脚本。

依次运行：

```powershell
# 训练受欢迎度模型（可选，建议启用 log1p 计数特征）
(venv) PS E:\旅游\backend> python -m app.pipelines.train_popularity_model --log-transform

# 基于内部评分计算内容受欢迎度（会按模型 meta 一致处理特征）
(venv) PS E:\旅游\backend> python -m app.pipelines.aggregate_content

# 基于评分行为生成/更新用户画像
(venv) PS E:\旅游\backend> python -m app.pipelines.aggregate_user_profile

# 基于评分构建 item-based 协同过滤相似度
(venv) PS E:\旅游\backend> python -m app.pipelines.aggregate_scenic_cf

# OTA 外部评分与评论数（CSV/样本）
(venv) PS E:\旅游\backend> python -m app.pipelines.load_ota_data

# 通过 CSV 快速导入 OTA/Social 样本（可选）
(venv) PS E:\旅游\backend> python -m app.pipelines.load_external_features E:\旅游\data\external_features_sample.csv

# 从 Wikipedia Pageviews 导入近30天浏览量（社交热度，限量200景点，可多次执行扩大覆盖）
(venv) PS E:\旅游\backend> python -m app.pipelines.fetch_wikipedia_pageviews --lang zh --limit 200 --days 30

# 覆盖率体检（查看 OTA/Social 覆盖与分布）
(venv) PS E:\旅游\backend> python -m app.pipelines.analyze_feature_coverage
```

可选：运行推荐效果评估脚本（用于实验和报告）：

```powershell
(venv) PS E:\旅游\backend> python -m app.pipelines.evaluate_recommendation
```

> 检查点（MySQL 示例；SQLite 可用 sqlite3 执行同样 SQL）：

```sql
USE tourism_imu;
SELECT * FROM content_standard
WHERE source_type IN ('internal_rating','ota_stats','social_media','item_cf_similarity')
LIMIT 10;
```

> 如能看到多条记录，说明多源数据已写入成功。

## 4. 系统演示完整流程

> 建议在答辩/展示时按此顺序演示，既能覆盖所有模块，又能突出“画像 + 多源数据 + 协同过滤 + 行程规划”的完整闭环。

### 4.1 用户注册与登录

1. 在前端导航栏点击“注册”：
   - 填写用户名（≥3 字符）、密码（≥6 字符）、邮箱（可选）。
   - 提交，看到“注册成功”的反馈。
2. 回到“登录”页面：
   - 使用刚注册账号登录。
   - 登录后导航栏显示“首页 / 浏览 / 我的偏好 / 行程规划”。

### 4.2 目的地浏览与评分（驱动画像与协同过滤）

1. 点击“浏览”：
   - 类型保持“景点”，城市可以留空或填写如“包头”“鄂尔多斯”。
   - 点击“搜索”，展示景点列表。
2. 在多条景点卡片上点击“我喜欢（打 5 分）”：
   - 弹出“感谢你的评分！推荐将逐渐向你的偏好靠拢。”
   - 这些评分会写入 `rating` 表，并被后续画像与协同过滤使用。
3. （可选）展示“查看相似景点”按钮：
   - 在任意景点卡片点击“查看相似景点”；
   - 卡片下方会展开“相似景点推荐”列表，展示由 item-based CF 计算出的邻居。

### 4.3 设置“我的偏好”（显式画像）

1. 点击“我的偏好”：
   - 页面已升级为**对话式偏好**：在输入框用自然语言描述偏好（目的地/天数/预算/兴趣/出行方式等）。
   - 示例输入：`想自驾去呼和浩特玩3天，预算3000，亲子`。
   - 发送后观察：
     - 左侧出现助手回复；
     - 右侧 `Slots`（识别结果）展示结构化字段；
     - 当识别到目的地后，右侧 `Itinerary` 自动生成行程草案预览；
     - 登录状态下，系统会按需写入 `user_profile`（如 `travel_style/budget_level/prefer_scenic_types` 等）。
   - 如需精确调整，可展开“手动编辑并保存”进行表单式微调。
2. 说明：
   - 后端推荐接口会同时考虑**显式偏好**与**基于评分的隐式画像**。
   - 如服务端未配置 `LLM_API_KEY`，对话将自动回退到本地规则抽取（可用但能力有限）；配置后会优先使用 LLM 做结构化抽取。

### 4.4 首页个性化推荐与多策略对比（热门 / 画像 / CF / MF / Hybrid）

1. 点击“首页 / 个性化推荐”：
   - “推荐景点”区域展示景点卡片；
   - 卡片上方有城市与评分信息。
2. 在“推荐景点”模块右上角的“策略”下拉框中，演示不同算法策略：
   - 选择“自动”：不显式指定策略，由后端根据是否有画像自动采用多源热门或画像重排；
   - 选择“热门（popular）”：只按多源综合得分（内部评分 + OTA + 社交媒体）排序，不考虑个体偏好；
   - 选择“画像（profile）”：在多源综合得分基础上优先展示符合 `user_profile.prefer_scenic_types` 的景点；
   - 选择“CF”：基于你对景点的评分记录，使用 item-based 协同过滤推荐与你喜欢景点“相似”的其他景点；
   - 选择“MF”：基于全体用户-景点评分训练一个简单矩阵分解模型，为你预测尚未评分景点的兴趣分数；
   - 选择“Hybrid”：在同一批候选景点上综合多源基线分 + CF 相似度得分 + MF 预测得分（权重约为 0.2 / 0.4 / 0.4），得到更平衡的个性化排序结果。
3. 每次切换策略后，点击“刷新”按钮：
   - 观察列表上方“当前策略：...”一行：
     - 若后端成功使用了对应算法（如 CF / MF / Hybrid），会显示 `cf` / `mf` / `hybrid`；
     - 若该用户评分数据过少导致 CF/MF/Hybrid 无法生效，后端会自动回退为热门或画像，对应标签也会显示回退后的策略；
     - 对于冷启动用户（评分和画像都很少），页面顶部会出现一条冷启动提示，引导用户先去“浏览”页多打几条分或在“我的偏好”中补充兴趣，以提升 CF/MF/Hybrid 的效果。
   - 对比不同策略下推荐列表的变化：
     - 热门：更多集中在评分高、评论多的头部景点；
     - 画像：列表中“符合你的偏好类型：XXX”的景点明显增多；
     - CF/MF：更容易出现你之前未评分、但和你喜欢景点“相似”的新景点；
     - Hybrid：在“个性化”与“整体口碑/热门度”之间取得折中，一般比单独 CF 或 MF 更稳健。
4. 继续强调“推荐理由”一行（绿色小字）：
   - 示例文案：
     - “本站评分较高”
     - “OTA 平台评分优秀/较好”
     - “社交媒体热度很高/讨论度较高”
     - “网友整体评价非常好/较好”
     - “符合你的偏好类型：草原风光”等。
   - 强调：这些理由来自多源数据与用户画像：
     - 内部评分聚合（`aggregate_content.py`）
     - OTA 数据（`backend/data/ota_sample.csv`，通过 `load_ota_data.py` + `OTACollector` 写入）
     - 社交媒体示例数据（`fetch_wikipedia_pageviews.py` / `load_external_features.py`，`social_media_fetcher.py` 为骨架）
     - 用户偏好设置与评分行为（`user_profile`、`rating`）。

### 4.5 行程规划演示

1. 点击“行程规划”：
   - 输入一个城市（如“包头”），点击加载景点；
   - 勾选 2–5 个景点作为本次出行目标。
2. 点击“规划路线”：
   - 页面展示推荐游览顺序；
   - 显示估算总里程数（km）。
3. 说明：
   - 后端 `POST /api/route/plan` 使用景点经纬度 + 最近邻启发式生成路线；
   - 若配置了 `AMAP_WEB_KEY` 环境变量，则会调用高德驾车路径规划 API 获取更精确距离。

### 4.6 空白新建行程演示

1. 点击“我的行程”：
   - 若已登录，列表页顶部会出现“新建空白行程”入口。
2. 点击“新建空白行程”：
   - 填写标题、出发城市、可选出发日期、预算等级、出游风格和天数；
   - 示例：`呼伦贝尔 4 天行程`，天数填 `4`。
3. 点击“创建空白行程”：
   - 页面跳转到 `/trips/:id` 详情页；
   - 可观察到系统已经生成 4 个 Day，但初始没有任何 item，这是一份合法的空草稿。
4. 在详情页中继续演示：
   - 先不新增任何 item，直接修改标题或某一天的备注并保存，说明空草稿可以直接保存 metadata；
   - 再在任意一天点击“新增条目”，补一个 `custom` 条目，填写标题和备注后保存；
   - 返回列表页，确认该行程的 `days`、`item_count` 和来源 `manual_draft` 显示正确。

### 4.7（可选）算法评估结果展示

> 若在答辩前跑过 `evaluate_recommendation.py` 并整理了指标结果，可在此简单展示：

1. 简述评估脚本做了什么（热门 vs 画像 vs 协同过滤 vs MF vs Hybrid）；
2. 展示 `precision@K` 等指标对比表或图，强调 Hybrid 在 Precision@5 上相对于 Popular / Profile / CF / MF 的表现；
3. 结合用户体验说明：
   - 在线系统看到的“推荐理由”“相似景点”“Hybrid 策略下的推荐列表变化”与离线评估指标之间的关系。

## 5. 常见问题与排查

1. **前端打不开 / 白屏**
   - 检查 `npm run dev` 是否正常运行；
   - 确认访问地址为 `http://localhost:5173/`。
2. **推荐接口报错 / 无推荐理由**
   - 确认已运行 2. 节中所有聚合和多源脚本；
   - 在数据库（SQLite/MySQL）中检查 `content_standard` 是否有 `internal_rating`、`ota_stats`、`social_media` 记录。
3. **相似景点为空**
   - 确认 `aggregate_scenic_cf.py` 已运行且 `rating` 中存在景点评分数据；
   - 检查 `content_standard` 中是否存在 `source_type = 'item_cf_similarity'` 的记录。
4. **行程规划结果为空**
   - 确认所选城市在 `scenic_spot` 表中有对应景点；
   - 确认提交时选择了至少 2 个景点。
5. **空白行程创建失败**
   - 确认已登录；
   - 确认 `title/origin_city/days` 已填写且 `days` 在 `1..10`；
   - 若后端返回校验错误，检查 `POST /api/trips` 请求体中的 `trip_days` 是否为合法数组。
