# 内蒙古旅游智能推荐与出行指南系统——项目讲解与答辩要点

> 本文档用于集中整理项目讲解稿、答辩要点和演示顺序。后续所有与“整体项目讲解/答辩讲稿”相关的内容，都在本文件上迭代更新，不再另建多个版本。

---

## 1. 项目整体介绍（1 分钟）

- **项目背景**
  - 围绕“内蒙古旅游”场景，存在景点分散、信息碎片化、游客决策困难等问题。
- **项目目标**
  - 利用多源数据 + 推荐算法，为游客提供吃、住、行、游一体化的智能推荐与出行指南。
  - 更强调系统的可用性和推荐质量，而不是只做展示界面。
- **核心亮点**
  - 多源数据融合：高德 POI（真实采集）+ 内部评分 + OTA 样本数据（`ota_stats`）+ 社交热度与情感（`social_media`）等。
  - 用户画像与个性化：显式偏好 + 行为评分驱动的 CF / MF / Hybrid 推荐。
  - 端到端闭环：从数据聚合、离线评估到在线接口和前端展示完整打通。

---

## 2. 系统架构与模块概览（1–2 分钟）

- **整体架构**
  - 前端：Vue 单页应用，包含首页推荐、目的地浏览、我的偏好、行程规划等页面。
  - 后端：Flask + SQLAlchemy，提供推荐、行程规划、用户与画像等 REST 接口。
  - 数据层：默认 SQLite（`dev.db`），亦支持 MySQL（通过 `DATABASE_URL`），存储用户、景点、酒店、美食、评分、多源内容标准化表等。
- **关键子系统**
  - 推荐子系统：热门/画像推荐、item-based CF、Biased MF、Hybrid 混合推荐。
  - 多源数据子系统：`content_standard` 聚合内部评分、OTA 指标（`ota_stats`）与社交热度/情感（`social_media`）。
  - 行程规划子系统：基于景点经纬度的简单路线规划，支持多景点顺序推荐。

> 讲解时可以结合《项目架构与实施规划.md》中的第 2 章简要展示架构图或文字版结构。

## 3. 推荐系统设计与实现亮点（2–3 分钟）

- **3.1 多源基线与画像推荐**
  - 说明多源基线分的构成：内部评分 + OTA 评分/评论数 + 社交媒体热度与情感。
  - 用户有画像时，在多源基线排序的基础上，对符合 `prefer_scenic_types` 的景点优先展示。
  - 这一部分也作为 CF/MF/Hybrid 的热门/画像基线和回退策略。

- **3.2 CF 与 Biased MF 个性化推荐**
  - CF：基于 `rating` 表构建 item-based 相似度，在 `content_standard` 中存储邻居及相似度。
  - Biased MF：在离线评估与在线接口中使用带全局均值、用户偏置、物品偏置的 MF 模型。
  - 强调：MF 不是孤立存在，而是与多源基线、CF 一起被纳入 Hybrid。

- **3.3 Hybrid 混合推荐策略**
  - 核心公式：在同一批候选上融合三路得分：
    - 多源基线分（内部+OTA+社交）；
    - CF 相似度得分；
    - Biased MF 预测评分；
  - 通道内做 min-max 归一化后，以 `w_base = 0.2, w_cf = 0.4, w_mf = 0.4` 加权，缺失通道自动重分配权重。
  - 在线默认策略：对非冷启动登录用户，未显式指定策略时自动使用 Hybrid。
  - 离线评估：在 `evaluate_recommendation.py` 中使用相同配方，对 Popular/Profile/CF/MF/Hybrid 做 Precision@5 对比；在合成/小规模评分数据下三者精度接近，Hybrid 主要体现“鲁棒性”（多通道互补 + 自动回退）。

- **3.4 冷启动与回退机制 + 监控**
  - 新用户或画像不足时自动识别 `cold_start`，回退到多源热门/画像策略，并在 `meta` 中记录 `fallback_reason`。
  - 对 CF/MF/Hybrid 推荐条数不足时，用多源热门结果做 backfill，并记录回退类型。
  - `/api/recommend/metrics` 暴露 `request_count`、`strategy_counts`、`fallback_counts`，前端首页高级设置中提供简易监控视图，方便答辩展示系统在真实使用中的策略分布与回退情况。

## 4. 前端演示流程与用户体验（2–3 分钟）

- 登录/注册 → 设置“我的偏好”（预算、景点类型、美食类型等）；

- 回到首页：
  - 首页提示说明系统会自动使用 Hybrid 等策略进行智能推荐；
  - “推荐景点”卡片展示推荐理由（多源数据 + 画像），并在高级设置中可切换策略用于实验；
  - “推荐酒店”“推荐美食”模块根据预算和美食偏好做排序；
- 行程规划页：选择若干景点，一键生成游玩顺序和大致里程，形成从推荐到出行规划的闭环。

## 8. 完整讲解稿示例（约 6 分钟）

> 下面是一份可以直接照读的讲解稿示例，可根据实际答辩时间和风格微调.

各位老师好，我汇报的题目是“内蒙古旅游智能推荐与出行指南系统”。这个系统主要是面向到内蒙古旅游的游客，希望解决两个痛点：

第一，信息非常分散。景点、美食、酒店的信息散落在各种平台上，游客很难快速做出决策；
第二，推荐比较粗糙。很多平台只是简单按热门排序，个性化程度不高，也很难结合出行预算和偏好给出一体化的出行建议.

围绕这两个问题，我们的目标是：基于多源数据和推荐算法，为游客提供“吃、住、行、游”一体化的智能推荐和出行指南。在实现上，我们更强调系统的可用性和推荐质量，而不是只做一个好看的前端页面.

接下来我简单介绍一下系统架构。整个系统分为三层：

第一层是前端展示层，采用 Vue 单页应用。主要包括：
- 首页的个性化推荐模块；
- “目的地浏览”页，可以查看景点列表并对景点评分；
- “我的偏好”页，用于设置出行风格、预算、偏好景点和美食类型；
- 以及行程规划页面，用于选择景点并生成游玩顺序.

第二层是后端服务层，基于 Flask + SQLAlchemy。它提供了一组 REST 接口，包括：景点、酒店、美食的推荐接口，用户与画像接口，以及行程规划接口等.

第三层是数据与算法层，主要使用 SQLAlchemy 连接数据库（默认 SQLite，亦支持 MySQL），存储用户、景点、酒店、美食、评分数据，以及一个叫做 `content_standard` 的标准化内容表，用来保存多源数据聚合后的指标，比如内部评分聚合、OTA 样本口碑（`ota_stats`）与社交热度/情感（`social_media`）等.离线的聚合脚本和评估脚本也都放在这一层.

在这个系统里，推荐子系统是核心亮点之一。我重点从三个方面说明：

第一部分是“多源基线和画像推荐”。
我们没有只用单一的内部评分，而是把三个来源融合到一起：
- 内部评分和评分次数，反映平台自身的口碑和热度；
- OTA 样本数据（人工整理真实页面）的外部评分和评论数，用于补充平台外部口碑信号；
- 社交热度/情感信号（如 Google Trends 等合法公开弱信号）的互动量和情感得分，用于补充网络讨论热度与好评度.

在此基础上，我们为每个景点构建了一个多源综合得分，作为热门推荐和后续算法的基线。同时，如果用户在“我的偏好”里设置了偏好景点类型，例如草原、湖泊等，我们会在候选列表中优先把这些类型的景点排在前面，这就是画像重排.

第二部分是 CF 和 Biased MF 的个性化推荐.
协同过滤这块，我们采用的是基于景点的 item-based CF。离线脚本会从评分表中构建用户-景点矩阵，计算景点之间的相似度，然后把每个景点的邻居列表写入 `content_standard` 表.在线上，当用户对一些景点打过分之后，我们可以从这些景点的相似邻居里给出“你可能也喜欢”的推荐.

矩阵分解方面，我们实现的是带全局均值、用户偏置、物品偏置的 Biased MF。在线上，为了保持实现简单，我们在请求时会基于当前评分数据训练一组轻量级的用户向量和景点向量，用统一的公式 `global_mean + b_u + b_i + <p_u, q_i>` 预测用户对未评分景点的兴趣.

第三部分是 Hybrid 混合推荐策略，这是本项目里一个比较重要的设计.
简单来说，Hybrid 并不是某一个新的模型，而是把三路得分融合在一起：
- 多源基线分数，保证推荐结果整体口碑和热门度不错；
- CF 相似度得分，保证推荐能贴近用户历史兴趣；
- MF 预测得分，用隐因子挖掘更深层的兴趣结构.

在同一批候选景点上，我们先分别计算这三路分数，并在候选集上做 min-max 归一化，之后按 `w_base = 0.2, w_cf = 0.4, w_mf = 0.4` 加权融合。如果某一路在当前候选里没有有效信号，比如 CF 找不到邻居，那么这一通道的权重会自动分配给剩余通道，保证数值稳定.

在在线策略上，我们把 Hybrid 作为有历史行为用户的默认策略：对已经登录且不是冷启动的用户，如果前端没有显式指定策略，后端会自动选择 Hybrid，结合多源、CF 和 MF 给出个性化排序.对于新用户或者画像不足的用户，系统会识别为冷启动，自动回退到多源热门或画像推荐，并在接口的 `meta` 信息中说明回退原因.

为了验证这些策略的效果，我们还实现了一个离线评估脚本 `evaluate_recommendation.py`.它会从评分表中取出景点评分，对每个用户按时间排序，用最后一条作为测试，其余作为历史行为，然后对 Popular、Profile、CF、MF 和 Hybrid 五种策略计算 Precision@5。在当前的合成/小规模评分数据上，CF、MF 和 Hybrid 的 Precision@5 大约都是 0.20，略高于 Popular 和 Profile.虽然在合成/小规模数据上三者数值非常接近，但 Hybrid 的优势主要体现在鲁棒性：在真实场景中，它能综合多源信息和行为信息，对冷启动和数据稀疏情况更稳健.

系统层面，我们还做了一些监控和可解释性设计.比如：
- 后端在每次推荐时都会记录实际使用的策略和回退原因，并在 `/api/recommend/metrics` 接口里统计 `request_count`、各策略使用次数以及回退次数；
- 前端首页的高级设置里，有一个小的“推荐统计”视图，可以看到在实际使用中 Hybrid、popular、profile 等策略的分布情况，以及各种回退发生的次数，这有助于我们在真实数据下调参.

前端演示时，我们会按这样一个顺序来走：
第一步，注册并登录一个用户，在“我的偏好”页面设置出行风格、预算等级、偏好景点类型和偏好美食类型.页面下方也会提示这些偏好将如何影响首页的景点、酒店和美食推荐.

第二步，进入“目的地浏览”页面，随机给几个景点打分.这样一方面可以更新用户画像，另一方面也为 CF、MF 和 Hybrid 提供必要的行为数据.

第三步，回到首页，展示“推荐景点”、“推荐酒店”和“推荐美食”三个模块.此时系统会根据用户的评分和偏好，默认使用 Hybrid 等策略给出智能推荐.景点卡片上会展示一些推荐理由，比如内部评分高、OTA 评分好、社交媒体热度高、或者“符合你的偏好类型：草原风光”等，让推荐结果更加可解释.酒店和美食模块则会结合预算等级和美食偏好对结果进行排序.

第四步，进入行程规划页面。在这里可以展示两种使用方式：
- 方式一：直接从首页推荐的景点卡片点击"加入行程"按钮，系统会自动将景点加入待规划列表；进入行程规划页面后，这些景点会自动加载并勾选，用户可以直接点击"规划路线"生成游玩顺序和大致里程；
- 方式二：在行程规划页面手动选择城市并加载景点列表，从中勾选若干景点进行规划.

这样可以向老师展示，从"画像与推荐"到"一键加入行程"再到"路线规划"的完整闭环，体现了推荐与出行指导的深度集成.

最后做一个简要总结：本项目从数据聚合、用户画像、推荐算法、行程规划，到前后端联调和监控，完整实现了一个面向内蒙古旅游场景的智能推荐与出行指南系统.算法方面，我们不仅实现了热门和画像推荐，还在此基础上接入了 item-based CF、Biased MF 以及基于多源数据的 Hybrid 混合策略，并通过离线评估和在线统计来验证效果和稳定性.后续如果有更多真实 OTA 或社交平台数据接入，我们可以在现有框架上继续扩展，更好地服务真实的旅游用户.

## 9. 与立项书主要内容的对应关系（讲解提示）

> 本节用于在答辩讲解时快速回答“你们和申报书里说的一致吗？”这一类问题，可结合立项书原文灵活发挥。

- **多源数据与数据库**
  - 立项书写法：强调整合社交媒体、电商、OTA、交通出行大数据、官方统计和文化习俗等多源异构数据，构建内蒙古旅游知识库与标准化数据库。
  - 当前实现：本系统已经有景点/酒店/美食等核心实体表，并通过高德 POI（真实采集）+ `content_standard` 聚合内部评分、OTA 样本口碑（`ota_stats`）与社交热度/情感（`social_media`，可由 Google Trends 等导入），支撑热门推荐、多源基线与可解释性；大规模真实 OTA/UGC 与交通大数据在合规与资源允许时可按同一管道扩展。
  - 讲解时建议：可以说“我们在工程上已经把多源数据的结构、口径与 ETL 流程打通，并且用可追溯的真实来源数据跑通链路；后续接入更大规模平台数据可复用现有框架。”

- **用户画像与行为分析**
  - 立项书写法：提出基于多源数据构建多维度用户画像与行为预测模型，对不同地域、年龄、性别等群体进行精细化刻画。
  - 当前实现：已经有 `user_profile` 表和“我的偏好”页面，支持出行风格、预算等级、景点类型、美食类型等显式偏好配置，并与评分行为一起驱动 CF/MF/Hybrid；同时已提供 `POST /api/behaviors` 行为上报与 `analyze_behavior.py` 分析脚本，形成可观测闭环。
  - 讲解时建议：强调“在当前数据条件下，我们优先把画像与推荐闭环打通，深度行为挖掘和预测模型会作为后续扩展方向，在现有日志与特征规划上继续推进。”

- **混合推荐模型（Hybrid）**
  - 立项书写法：提出融合协同过滤、内容过滤与深度神经网络的混合推荐模型，用于缓解数据稀疏和冷启动问题。
  - 当前实现：工程上实现了“多源内容特征 + item-based CF + Biased MF”的 Hybrid 混合策略，并作为非冷启动用户的默认策略；同时在离线评估脚本中对 Popular / Profile / CF / MF / Hybrid 做了 Precision@5 对比。
  - 讲解时建议：可以说明“我们已经用 CF + Biased MF + 多源内容实现了一个可落地的混合推荐模型，深度神经网络部分在当前数据规模和时间下暂未上线，会作为下一阶段算法深化的自然延伸。”

- **动态路线规划与系统原型**
  - 立项书写法：结合实时交通状况，为游客动态生成‘吃、住、行、游’一体化路线规划与出行指导。
  - 当前实现：系统已经支持基于景点经纬度的行程规划，并预留了高德地图路线规划 API 接入点；首页推荐与行程规划在产品流程上已经串联起来，但尚未真正引入大规模实时交通数据。
  - 讲解时建议：可以把当前功能定位为“路线规划原型”，说明“我们已经打通了从推荐到行程的流程，未来在接入真实交通数据后可以在此基础上做动态优化”。
