## 三、技术实现

### 3.1 采集架构

```
数据源 → 采集器 → 解析器 → 数据库
  ↓        ↓        ↓        ↓
高德API  AmapCollector  字段映射  MySQL
OTA平台  OTACollector   标准化   content_standard
```

### 3.2 核心组件

**1. 基础采集器 (BaseCollector)**
- 统一采集接口
- 速率限制机制
- 错误处理与重试
- 数据验证

**2. 高德采集器 (AmapCollector)**
- POI搜索与分页
- 动态字段解析（处理列表/字符串类型）
- 坐标转换
- 图片URL提取

**3. OTA采集器 (OTACollector)**
- CSV数据读取
- API接口预留
- 字段标准化

**4. 数据匹配器 (DataMatcher)**
- 模糊匹配算法（difflib.SequenceMatcher）
- 相似度阈值: 0.8
- 基于名称+城市+地址的三重匹配

### 3.3 ETL流程

**Extract (提取)**:
- API调用 / CSV读取
- 分页处理
- 速率控制

**Transform (转换)**:
- 字段映射
- 类型转换
- 数据清洗
- 去重处理

**Load (加载)**:
- 实体匹配
- 数据融合
- 数据库插入/更新
- 聚合到content_standard表

## 五、数据融合

### 5.1 融合策略

**实体匹配算法**:
```python
similarity = SequenceMatcher(None, name1, name2).ratio()
if similarity >= 0.8 and city_match and address_similar:
    match_success = True
```

**人气分数计算**:
```python
popularity = (rating / 5.0) * 60 + min(review_count / 100, 40)
# 最高100分，评分占60%，评论数占40%
```

### 5.2 融合结果

- **总匹配数**: 4/70 (5.7%)
- **成功案例**: 
  - 五当召（包头）
  - 香格里拉大酒店（呼和浩特）
  - 希拉穆仁草原（包头）
  - 阿尔山国家森林公园（兴安盟）

**匹配率较低原因分析**:
1. 高德数据与OTA数据命名差异较大
2. OTA数据包含更多不在高德数据库中的POI
3. 相似度阈值设置较严格（0.8）

**改进方向**:
1. 降低相似度阈值至0.6-0.7
2. 增加别名映射表
3. 引入外部ID映射

## 七、技术挑战与解决

### 7.1 遇到的问题

**问题1: API速率限制**
- 现象: `CUQPS_HAS_EXCEEDED_THE_LIMIT`
- 解决: 调整速率限制从0.1秒增至0.5秒

**问题2: 动态字段类型**
- 现象: `tel`、`address`等字段为列表
- 解决: 添加类型检查和转换逻辑

**问题3: 数据库字段映射**
- 现象: `type`字段不存在
- 解决: 修正为`category`字段

**问题4: MySQL连接失败**
- 现象: 数据库服务未启动
- 解决: 启动MySQL80服务

### 7.2 优化措施

1. **并发控制**: 严格遵守API速率限制
2. **容错处理**: 跳过无效数据，记录警告日志
3. **增量更新**: 支持数据更新，避免重复插入
4. **批量提交**: 减少数据库IO次数
