# 内蒙古旅游智能推荐与出行指南系统 - 数据库设计

> 本文档用于描述本项目的数据库概念模型与表结构设计，是数据库相关工作的唯一权威文档。任何后续表结构调整，都应在本文件中同步更新，并记录于“更新记录”章节。

---

## 1. 设计目标与原则

- **支持核心业务流程**：用户注册登录 → 浏览/收藏/评价 → 获取推荐 → 生成与调整行程 → 路线规划。
- **支持多源数据融合**：可承载社交、电商、OTA、官方等多源数据的标准化存储。
- **兼顾分析与推荐**：既满足在线业务访问，又方便离线画像与推荐模型训练。
- **易于扩展与维护**：命名清晰、适度规范索引，避免过早优化但保留扩展空间。

命名说明（建议）：
- 表名：使用小写+下划线，如 `user`, `scenic_spot`, `user_profile` 等。
- 主键：优先使用自增整型 `id`，必要时可辅以业务唯一键（如 `username`）。
- 时间字段：统一使用 `created_at`, `updated_at` 形式，类型可用 `DATETIME`。

> 说明：本设计以 MySQL 为目标库；本地开发默认使用 SQLite（`dev.db`），字段与表结构保持兼容。

## 3. 用户与画像相关表设计

### 3.1 用户表 `user`

**用途**：存储系统登录用户的基础信息，是用户相关表的核心主表。

**字段示例（逻辑层面）**：
- `id`：主键，自增整数。
- `username`：用户名，唯一。
- `password_hash`：密码哈希。
- `email`：邮箱，可空。
- `phone`：手机号，可空。
- `gender`：性别枚举（未知/男/女）。
- `age`：年龄或年龄段标识。
- `home_region`：常住地区（省/市），可结合内蒙古外地用户来源分析。
- `register_source`：注册来源（Web/移动/第三方等）。
- `created_at`：创建时间。
- `updated_at`：更新时间。

### 3.2 用户画像表 `user_profile`

**用途**：存储经过画像算法计算后的用户标签和偏好特征，用于快速推荐。

**字段示例**：
- `id`：主键，对应 `user.id` 或一对一关系。
- `user_id`：用户外键。
- `cluster_id`：用户分群编号（如 K-means 聚类结果）。
- `prefer_scenic_types`：偏好景点类型标签列表（如草原/湖泊/人文等，序列或 JSON）。
- `prefer_food_types`：偏好美食类型标签。
- `travel_style`：出游方式（家庭/情侣/独自/团队等）。
- `budget_level`：消费水平或价格敏感度等级。
- `travel_frequency`：年均出游频次等统计特征。
- 规划字段（当前实现未落地）：`gender`、`age_group`、`region_code`、`origin_feature_json`。
  - 现阶段画像排序使用 `user` 表中的 `gender/age/home_region`，未在 `user_profile` 冗余存储。
- `feature_vector`：可选，保存特征向量（如降维后的特征，可用 JSON/BLOB）。
- `updated_at`：最近一次画像更新时间。

### 3.3 用户行为日志表 `user_behavior_log`

**用途**：记录用户在系统内的关键行为，为画像与推荐提供原始数据。

**字段示例**：
- `id`：主键。
- `user_id`：用户外键，可为空（未登录行为）。
- `target_type`：目标类型（景点/美食/特产/行程等）。
- `target_id`：目标 ID。
- `behavior_type`：行为类型（浏览、点击、收藏、下单、分享、评价等）。
- `behavior_value`：行为强度（例如停留时长、评分值等）。
- `device`：访问设备或终端信息。
- `ip`：IP 地址（可用于地域分析与反作弊）。
- `occurred_at`：行为发生时间。

> 实现建议：业务线上可只保留摘要行为，详细日志可选用独立日志系统或大数据平台，这里以满足课题与推荐算法需求为主。

## 4. 旅游资源相关表设计

### 4.1 景点表 `scenic_spot`

**用途**：存储内蒙古地区景点的基础信息和统计信息，是核心推荐对象之一。

**字段示例**：
- `id`：主键。
- `name`：景点名称。
- `city`：所在城市/盟市。
- `district`：区/旗。
- `address`：详细地址。
- `longitude` / `latitude`：经纬度坐标。
- `type`：景点类型（草原/湖泊/沙漠/人文景观等）。
- `tags`：标签（适合人群、特色等，JSON 或字符串）。
- `opening_hours`：开放时间。
- `ticket_price`：门票价格区间或参考价。
- `rating_avg`：平均评分（来自用户评价汇总）。
- `rating_count`：评分次数。
- `description`：文字介绍。
- `images`：图片链接列表。
- `status`：状态（正常/下线等）。
- `created_at` / `updated_at`：时间字段。

### 4.2 美食/餐饮表 `food_place`

**用途**：存储美食与餐厅信息，用于“吃”的推荐。

**字段示例**：
- `id`：主键。
- `name`：店名或美食名称。
- `city` / `district`：所在位置。
- `address`：详细地址。
- `longitude` / `latitude`：坐标（便于地图显示）。
- `cuisine_type`：菜系/类型（烧烤、手扒肉、奶制品等）。
- `avg_price`：人均消费。
- `rating_avg` / `rating_count`：评分信息。
- `tags`：标签（适合夜宵、适合家庭等）。
- `source`：数据来源（美团/大众点评/人工录入等）。
- `created_at` / `updated_at`。

### 4.3 住宿表 `hotel`

**用途**：存储酒店/民宿信息，用于“住”的推荐。

**字段示例**：
- `id`：主键。
- `name`：住宿名称。
- `city` / `district`。
- `address`。
- `longitude` / `latitude`。
- `star_level`：星级或档次标签。
- `avg_price`：参考价格。
- `rating_avg` / `rating_count`。
- `tags`：例如亲子友好、靠近景区等。
- `source`：数据来源。
- `created_at` / `updated_at`。

### 4.4 特产/礼品表 `souvenir`

**用途**：存储与内蒙古相关的特产/礼品商品信息，支持礼品/伴手礼推荐。

**字段示例**：
- `id`：主键。
- `name`：商品名称。
- `category`：类别（奶制品、肉制品、饰品等）。
- `origin_city`：主要产地。
- `price`：参考价格或价格区间。
- `rating_avg` / `rating_count`：电商平台/用户评分。
- `sales_volume`：销量统计（可从电商平台统计）。
- `tags`：适合作为礼品/自用等标签。
- `image`：主图链接。
- `source`：数据来源（淘宝/天猫/线下整理等）。
- `created_at` / `updated_at`。

## 5. 行程与评价相关表设计

### 5.1 行程主表 `trip`

**用途**：存储用户的整体出行计划。当前已同时支持：
- 从 `/route` 保存单日路线结果；
- 从 `/profile` 的 Agent 草案保存多日行程；
- 从 `/trips/new` 直接创建空白草稿；
- 在“我的行程”中统一编辑 1~10 天的日程结构。

**字段示例**：
- `id`：主键。
- `user_id`：用户外键。
- `title`：行程名称（可自动生成，也可用户自定义）。
- `start_date` / `end_date`：出行起止日期。
- `days`：出行天数（冗余字段，按 `trip_day` 数量归一化计算，便于查询）。
- `origin_city`：出发城市。
- `budget_level`：预算水平。
- `travel_style`：出游方式（家庭/情侣/团队等）。
- `created_by`：创建来源（用户手动/系统推荐）。
  - 当前已使用的来源值包括 `route_planner`、`agent_planner`、`manual_draft`。
- `created_at` / `updated_at`。

### 5.2 行程日表 `trip_day`

**用途**：行程按照天拆分的维度，用于多日行程管理。

**字段示例**：
- `id`：主键。
- `trip_id`：行程外键。
- `day_index`：第几天（1,2,3...）。
- `date`：具体日期（可选）。
- `note`：当日备注。

### 5.3 行程明细表 `trip_item`

**用途**：记录每天的具体活动（景点、美食、交通等），支撑时间轴展示与路线规划。

**字段示例**：
- `id`：主键。
- `trip_day_id`：行程日外键。
- `item_index`：当日顺序（1,2,3...）。
- `item_type`：类型（scenic/food/hotel/traffic/other）。
- `ref_id`：对应资源表主键（如景点 id）。
- `title_snapshot`：保存规划当时的标题快照，避免源数据后续变更导致行程不可读。
- `city_snapshot`：保存规划当时的城市快照。
- `address_snapshot`：保存规划当时的地址快照。
- `start_time` / `end_time`：计划时间。
- `note`：备注（例如“预留午休时间”）。
- `transport_mode`：可选，交通方式（walk/bus/metro/drive/taxi）。

### 5.4 评价表 `rating`

**用途**：存储用户对资源或行程的评分和文字评价。

**字段示例**：
- `id`：主键。
- `user_id`：用户外键。
- `target_type`：评价对象类型（景点/美食/住宿/行程等）。
- `target_id`：对象 ID。
- `score`：评分（如 1–5）。
- `comment`：文字评价。
- `created_at`：评价时间。

## 6. 多源数据与标准化表设计

本节定义用于承载外部平台原始数据和标准化内容的表结构。实际字段可按具体平台接口细化。

### 6.1 社交媒体原始数据表 `social_raw`

**用途**：存储微博/小红书/短视频平台等的原始帖子和互动数据。

**字段示例**：
- `id`：主键。
- `platform`：平台标识（weibo/xiaohongshu/douyin 等）。
- `post_id`：平台原始帖子 ID。
- `user_name` / `user_id`：平台用户信息（可脱敏）。
- `content`：文本内容。
- `images`：图片链接列表。
- `likes` / `comments` / `shares`：互动统计。
- `publish_time`：发布时间。
- `crawl_time`：爬取时间。

### 6.2 电商平台原始数据表 `ecommerce_raw`

**用途**：存储淘宝/天猫等平台关于特产/礼品的商品数据。

**字段示例**：
- `id`：主键。
- `platform`：平台标识。
- `product_id`：商品 ID。
- `title`：商品标题。
- `price`：价格。
- `sales_volume`：销量。
- `rating_avg` / `rating_count`：评分数据。
- `shop_name`：店铺名。
- `crawl_time`：爬取时间。

### 6.3 标准化内容表 `content_standard`

**用途**：将多源数据处理后统一映射到“景点/美食/特产等实体”的内容表，作为推荐的输入之一。

**字段示例**：
- `id`：主键。
- `entity_type`：实体类型（`scenic_spot`/`food_place`/`hotel`，`souvenir` 规划保留）。
- `entity_id`：实体主键。
- `source_type`：来源类型（`internal_rating` / `ota_stats` / `social_media` / `item_cf_similarity` 等，兼容历史 `ota`）。
- `title` / `summary`：摘要信息。
- `sentiment_score`：情感分析得分。
- `popularity_score`：综合热度评分（基于互动量、销量等）。
- `last_update`：最后更新日期。

## 7. 索引、性能与扩展建议

- 为高频查询字段建立索引
  - 如 `user_behavior_log.user_id`, `scenic_spot.city`, `trip.user_id` 等。
- 避免过度规范化导致查询复杂
  - 在必要处保留冗余字段（如 `trip.days`）。
- 预留扩展字段
  - 可用 JSON 字段存储难以提前确定的标签和属性。

后续如在实际开发中发现性能瓶颈，可在本节追加针对性优化（如分库分表、缓存策略等）。
