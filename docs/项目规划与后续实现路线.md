## 1. 当前基础版本回顾（起点）

当前代码与系统已经实现的核心能力可概括为：

1. **端到端 Web 原型系统**
   - 后端：Flask + SQLAlchemy（默认 SQLite，亦支持 MySQL，通过 `DATABASE_URL` 切换），提供统一 `/api/...` 接口；
   - 前端：Vue 3 + Vite 单页应用，包含推荐、目的地浏览、用户登录/注册、偏好设置、行程规划等页面；
   - 用户可以从注册登录 → 浏览景点/酒店/美食 → 打分 → 查看推荐 → 规划行程，形成一个完整体验闭环。

2. **内部数据与推荐算法雏形**
   - 基于自建数据库的评分和行为数据，已实现：
     - 热门推荐（评分 + 人气排序）；
     - 基于 `user_profile` 的类型偏好推荐；
     - item-based 协同过滤相似度与“相似景点推荐”；
     - 简单行程规划（最近邻 + 经纬度估算）。

3. **架构与文档体系**
   - 总体规划：`项目架构与实施规划.md`；
   - 接口与模块说明：`docs/接口与模块设计.md`；
   - 多源数据规划：`docs/数据采集与处理方案.md`；
   - 阶段进展总结：`docs/阶段进展与已完成功能总结.md`。

后续工作将以此为基础，重点突破“大规模真实数据获取与分析”和“算法效果评估”两大方向。

## 2. 多源数据采集与分析规划

### 2.1 短期（可在当前开发环境内完成）的目标

1. **至少打通一个真实数据源**
   - 备选：
     - 社交媒体平台提供的开放 API 或示例数据集；
     - OTA 平台导出的 CSV/Excel 示例（课程或老师提供的样例数据）；
   - 要求：能从外部数据 → 清洗 → 映射到本地 `scenic_spot` / `content_standard`，并被推荐模块调用到。

2. **扩展现有骨架脚本**
   - 在 `app/pipelines/social_media_fetcher.py` 中补充：
     - 实际的请求逻辑（如果有公开 API）或对现有 JSON/CSV 示例数据的读取；
     - 基于文本与景点名称的匹配策略，将社交媒体内容聚合到景点级指标上（如近 30 天内容数、互动数、简单情感均值）。
   - OTA 数据采集部分（已完成首版）：
    - 使用 `backend/data/ota_sample.csv` 作为真实 OTA 数据源（70 条）；
    - 通过 `OTACollector` + `load_ota_data.py` 完成“读取 CSV → 匹配实体 → 写入 content_standard(source_type="ota_stats")（兼容历史 ota）”的 ETL 流程；
    - 后续若需要更多 OTA 平台或更大规模数据，只需扩展 CSV 内容或在 `OTACollector._fetch_from_api` 中接入真实 API。

  - 字段约定（与推荐模块对齐）：
    - `entity_type`：`scenic_spot | hotel | food_place`（兼容历史 `food`）；
    - `summary`：读取时兼容 `external_rating/rating` 与 `review_count/external_review_count`。

3. **形成一个可复现的数据集快照**
   - 在 `data/processed/` 下保存：
     - 清洗后的景点评分/行为数据集（可用于离线训练和评估）；
     - 外部数据的已映射版本（例如 `processed_ota_scenic.csv`）；
   - 在文档中说明数据字段含义、来源与使用方式，便于他人复现实验。

### 2.2 中长期规划

1. **更多数据源与特征**
   - 社交媒体：
     - 引入更细粒度的主题标签、情感分析结果；
     - 识别“打卡景点”、“冷门高口碑景点”。
   - 交通与天气：
     - 加入节假日客流、实时拥堵指数、天气状况等动态变量，影响推荐与行程规划。

2. **数据质量与治理**
   - 建立异常值检测、缺失值处理、标准化与归一化流程；
   - 对关键特征（评分、互动数、热度指数）做统计分析，形成可视化报表和描述性统计。

3. **统一特征仓库**
   - 在 `content_standard` 与 `user_profile` 的基础上，梳理一个清晰的“特征字典”：
     - 每个字段的含义、来源（内部/外部）、更新频率、用于哪些模型或接口；
   - 便于后续引入更复杂的模型（例如学习排序、深度推荐）。

## 3. 推荐算法与效果评估规划

### 3.1 现有算法基线

- 热门推荐：按平均评分与评分次数排序；
- 基于画像的类型偏好推荐：利用 `user_profile.prefer_scenic_types` 调整排序；
- 协同过滤：item-based CF + 余弦相似度，提供相似景点推荐；
- 行程规划：基于经纬度的最近邻启发式，并预留高德路径规划优化。

这些构成了后续实验中的 **Baseline 1（热门）**、**Baseline 2（画像）**、**Model 1（协同过滤）** 与 **Route 1（启发式规划）**。

### 3.2 计划引入的算法与模型

1. **混合推荐策略**
   - 将热门、画像和协同过滤结果按一定权重融合：
     - 例如：`score_final = w_pop * score_pop + w_profile * score_profile + w_cf * score_cf`；
   - 在实验中调参，比较不同权重配置对点击率/满意度的影响。

2. **排序模型（可选）**
   - 使用梯度提升树（如 XGBoost）或逻辑回归，对候选集进行二次排序；
   - 特征来源：用户画像、景点特征、多源数据指标、历史点击/评分情况；
   - 需要准备训练集（点击/未点击或高评分/低评分）与测试集。

3. **时间序列与动态推荐（如果时间允许）**
   - 加入时间维度，分析景点在不同季节、月份的热度变化；
   - 对节假日提前给出不同的推荐策略和行程规划建议。

### 3.3 效果评估脚本与指标

1. **评估脚本规划**
   - 在 `backend/app/pipelines/` 或 `backend/eval/` 下新增评估脚本，例如：
     - `evaluate_recommendation.py`：
       - 随机划分训练集/测试集；
       - 对热门、画像、协同过滤及混合策略计算 `precision@K`、`recall@K`、`MAP` 等指标；
       - 输出结果到 `data/processed/eval_results.json` 或 CSV。

2. **实验设计**
   - 离线实验：
     - 基于历史评分/点击日志，模拟用户的真实偏好；
     - 对每种策略分别计算指标，并对比优劣。
   - 在线/半在线实验（如时间允许）：
     - 在前端开放两个入口或参数，随机分配用户使用不同推荐策略；
     - 对比一段时间内的点击率、停留时间、评分行为等。

3. **结果呈现与论文支持**
   - 将评估结果整理为表格和图表（折线图/柱状图）；
   - 在论文中结合用户调查或问卷结果，给出综合结论。

## 4. 前端与用户体验后续优化

1. **用户中心与历史记录**
   - 增加“我的行程”、“我的评分”、“浏览历史”等页面；

2. **地图可视化**
   - 在行程规划页中，集成地图渲染（调用高德 JS API 或 Vue 组件），将推荐路线绘制在地图上；
   - 提示关键路段距离与预计耗时。

3. **推荐解释与可解释性展示**
   - 在推荐卡片上显示简单的“推荐理由”：
     - 例如：“你喜欢的类型：草原”，“与你评分较高的景点相似”，“当前城市热门景点”等；
   - 有利于提高用户信任感，也便于答辩时展示系统的“智能性”。
