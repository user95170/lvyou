## 3. API 设计约定

- 协议：HTTP + JSON
- 成功响应：通常为 `200` 或 `201`，返回结构为：
  - 列表类：`{"items": [...], "pagination" 或 "meta": {...}}`

- 失败响应：
  - 返回形如：`{"error": "错误说明"}`，状态码常为 `400` / `404` 等

## 7. 行为日志与评分接口

### 7.1 行为日志

- 路径：`POST /api/behaviors`
- 用途：记录浏览、点击、收藏等行为，为推荐与画像提供数据
- 请求体示例：

```json
{
  "user_id": 1,                   // 可选，未登录可为空
  "target_type": "scenic_spot",  // 必填：scenic_spot/hotel/food_place 等
  "target_id": 3,                 // 必填
  "behavior_type": "view",       // 必填：view/click/favorite...
  "behavior_value": 1,            // 可选：数值型，如停留时长
  "device": "web",               // 可选
  "ip": "127.0.0.1"              // 可选，不填则默认请求 IP
}
```

### 7.2 评分评价

- 路径：`POST /api/ratings`
- 用途：对景点/酒店/美食打分并评论，同时更新目标实体的平均分与评分数
- 请求体示例：

```json
{
  "user_id": 1,
  "target_type": "scenic_spot",   // scenic_spot/hotel/food_place
  "target_id": 3,
  "score": 5,                       // 1~5
  "comment": "很不错的草原体验"
}
```

- 成功响应：包括本条评分和聚合结果：

```json
{
  "rating": { ...单条评分... },
  "aggregate": {
    "rating_avg": 4.8,
    "rating_count": 12
  }
}
```

## 8. 推荐接口

### 8.1 景点推荐（支持个性化）

- 路径：`GET /api/recommend/scenic-spots`
- 查询参数：
  - `city`：可选，按城市过滤
  - `limit`：返回条数，默认 10，上限 50
  - `user_id`：可选，提供时会尝试根据画像做轻量个性化

行为说明：

- 无 `user_id` 或无画像：
  - 按城市等条件筛选景点；
  - 先按内部评分与评分人数做一轮粗排；
  - 再基于多源数据计算综合得分：
    - 内部评分聚合得到的 `popularity_score`（`source_type = "internal_rating"`），该分数由脚本 `aggregate_content.py` 调用受欢迎度回归模型 `popularity_model.pkl` 根据 `rating_avg`、`rating_count` 等特征预测；若模型文件不存在或加载失败，聚合脚本会回退为基础公式（`rating_avg * (1 + ln(1 + rating_count))`），以保证可用；如需更精确效果再训练模型；
    - OTA 外部评分与评论数（`source_type = "ota_stats"`，可由 `load_ota_data.py` 或 `load_external_features.py` 写入 `summary.external_rating` 与 `summary.review_count` 等字段）；
    - 社交媒体互动与情感（`source_type = "social_media"`，从 `social_media_fetcher.py` 或 `load_external_features.py` 写入的 `summary.interaction_sum`、`summary.sentiment_avg`）；
  - 综合得分示意：`内部受欢迎度 + 外部评分 + 社交热度 + 情感加权`，综合分越高排序越靠前。
- 有画像（提供了 `user_id` 且存在 `UserProfile.prefer_scenic_types`）：
  - 从 `user_profile.prefer_scenic_types` 中取得偏好景点类型集合；
  - 在上述多源综合得分的基础上，引入“类型优先”规则：
    - 先将类型属于偏好集合的景点排在前面；
    - 在偏好类型内部，继续按多源综合得分与评分、人气等排序。

> 小结：景点推荐接口实现了“画像 + 内部评分 + OTA 外部指标 + 社交媒体热度与情感”的多源融合排序，在保证兼容老接口的同时，让推荐结果更贴近真实热度与游客口碑。

### 8.2 相似景点推荐（协同过滤）

- 路径：`GET /api/recommend/scenic-spots/{spot_id}/similar`
- 查询参数：
  - `limit`：返回条数，默认 10，上限 50
- 行为说明：
  - 依赖离线脚本 `aggregate_scenic_cf.py` 生成的 item-based 协同过滤相似度；
  - 对于指定景点 `spot_id`，从 `content_standard` 中读取相似邻居列表：
    - `entity_type = "scenic_spot"`
    - `source_type = "item_cf_similarity"`
  - 按协同过滤相似度 + 评分 + 人气排序后返回前 N 个相似景点。

### 8.3 酒店推荐

- 路径：`GET /api/recommend/hotels`
- 行为说明：
  - 多源综合排序：
    - 内部评分聚合得到的 `popularity_score`（`source_type = "internal_rating"`，由 `aggregate_content.py` 通过 ML 模型预测）；
    - OTA 指标（`source_type = "ota_stats"`，外部评分 `external_rating`、评论数 `review_count` 等）；
    - 社交媒体指标（`source_type = "social_media"`，`interaction_sum`、`sentiment_avg` 等）。
  - 内部回退：当缺少 `internal_rating` 聚合结果时，在线按 IMDb 风格 Bayesian WR 即时计算内部基线分（实体类型独立的 `C/m` 参数）。
  - 画像可用时：按预算段等轻量特征优先，再在集合内部按综合分、评分与人气排序。

### 8.4 美食推荐

- 路径：`GET /api/recommend/foods`
- 行为说明：
  - 多源综合排序：
    - 内部评分聚合得到的 `popularity_score`（`source_type = "internal_rating"`，由 `aggregate_content.py` 通过 ML 模型预测）；
    - OTA 指标（`source_type = "ota_stats"`，外部评分与评论数）；
    - 社交媒体指标（`source_type = "social_media"`，互动与情感）。
  - 内部回退：缺少 `internal_rating` 时，按 Bayesian WR 即时计算内部基线分（实体类型独立的 `C/m` 参数）。
  - 画像可用时：按菜系/标签等轻量特征优先，再在集合内部按综合分、评分与人气排序。

> 说明：酒店与美食推荐也可以在后续版本中接入用户画像（如偏好菜系、酒店档次），当前保留为热门榜接口。

## 9. 数据聚合与画像管道

### 9.1 内容受欢迎度聚合

- 脚本：`app/pipelines/aggregate_content.py`
- 相关训练脚本：`app/pipelines/train_popularity_model.py`
- 用途：
  - 基于 `ScenicSpot`、`Hotel`、`FoodPlace` 的评分统计，优先加载机器学习回归模型 `popularity_model.pkl`，预测 `popularity_score` 并写入 `content_standard`，`source_type = "internal_rating"`；若模型文件缺失或加载失败，脚本会回退到基础公式计算以保证可用；
  - 受欢迎度模型由 `train_popularity_model.py` 采用多源特征训练（内部评分均值/次数、OTA 指标、社交指标），以“Bayesian WR + OTA + 社交热度/情感”的组合为监督信号；优先使用 LightGBM（启用单调约束），环境不可用时回退 RandomForest；
  - 训练端可通过 `--log-transform` 对计数类特征（如 `rating_count/ota_review_count/social_interactions`）做 log1p 变换；训练完成后会在 `app/pipelines/models/popularity_model.meta.json` 中保存变换与权重等元信息；聚合时 `aggregate_content.py` 会读取该元信息并一致应用，以保证训练/预测两端特征处理一致。
- 运行方式（在 `E:\旅游\backend` 下）：

```powershell
# 训练或更新受欢迎度模型（可按需执行，至少需要一条评分数据）
python -m app.pipelines.train_popularity_model

# 基于当前模型与评分统计计算 popularity_score
python -m app.pipelines.aggregate_content
```

### 9.3 协同过滤相似度聚合（item-based CF）

- 脚本：`app/pipelines/aggregate_scenic_cf.py`
- 用途：
  - 基于 `rating` 中景点评分记录构建 item-based 协同过滤相似度；
  - 为每个景点计算相似邻居列表，写入 `content_standard`：
    - `entity_type = "scenic_spot"`
    - `source_type = "item_cf_similarity"`
    - `summary` 为 JSON：`{"neighbors": [{"id": scenic_id, "sim": similarity}, ...]}`；
  - 供 `/api/recommend/scenic-spots/{spot_id}/similar` 接口读取使用。
- 运行方式（在 `E:\旅游\backend` 下）：

```powershell
python -m app.pipelines.aggregate_scenic_cf
```

## 12. 个性化电子地图（新增）

### 12.1 `POST /api/map/personalized`

- 功能：按用户画像与约束返回多要素（景点/美食/住宿/交通/活动/特产）的个性化列表。
- 请求体示例：

```json
{
  "user_id": 1,
  "city": "呼和浩特",
  "bounds": {"min_lng": 111.6, "min_lat": 40.7, "max_lng": 112.1, "max_lat": 41.1},
  "categories": ["scenic_spot", "food_place", "hotel"],
  "limit_per_type": 20,
  "filters": {"price": {"max": 200}, "cuisine": ["本地特色"]}
}
```

- 响应示例（节选）：

```json
{
  "items": {
    "scenic_spot": [{"id": 101, "name": "昭君博物院", "fit_reasons": ["人文历史", "安静舒适"]}],
    "food_place": [{"id": 501, "name": "内蒙特色手把肉", "fit_reasons": ["偏辣口味", "本地特色"]}],
    "hotel": [{"id": 801, "name": "市中心舒适酒店", "fit_reasons": ["安静", "交通便捷"]}]
  },
  "meta": {"strategy": "multi_source+profile-demographic", "city": "呼和浩特", "profile_used": true}
}
```

- 行为说明补充：
  - 当提供 `user_id` 且存在画像信息时，系统基于用户的性别（`gender`）、年龄（`age`→`age_group`）、常住地区（`home_region`）、偏好类型（`prefer_*`）、预算段（`budget_level`）进行轻量重排；
  - 未提供画像或画像缺失时，保持按内部评分与人气的默认排序；
  - 返回结构不变，仅结果顺序调整。

```json
  "itinerary": {

        {"type": "scenic_spot", "id": 101, "start_time": "09:30", "end_time": "11:30"},
        {"type": "food_place", "id": 501, "start_time": "12:00", "end_time": "13:00"}
      ]}
    ]
  }
}
```

- 行为说明补充：
  - 可选 `user_id` 时，候选集合会先按画像偏好进行重排（同个性化地图的轻量规则）；
  - 每天内部的项目顺序会按经纬度最近邻做路线顺序优化，尽量减少走回头路；
  - 若缺少坐标或画像，回退到原有的均衡分配与顺序逻辑。
  - `days` 默认 2，范围会被夹在 `1..10`；若传入非整数会返回 `400`。
  - `candidates` 若传入必须为对象，否则返回 `400`。
  - 当指定城市下景点与美食均为空（且未提供有效候选）时，返回 `400`：`{"error": "当前暂无可用内容，无法生成行程"}`。

## 14. 路线方案（新增）

### 14.1 `POST /api/route/options`

- 功能：给定起终点（可含途经点）与偏好，返回“最快/少换乘/步行优先”等方案。

- 请求参数：
  - `origin`、`destination`：对象，包含 `lng/lat`（必填）。
  - `mode`：`drive` / `transit` / `walk`（可选，默认 `drive`）。
- `city`：公交换乘模式建议提供，城市名称或 `citycode`（如北京 `010`）。若未提供，后端会在 `mode=transit` 时尝试基于起点/终点进行逆地理推断 `citycode`；跨城规划建议显式提供 `city`/`citycode`。
- `prefer`：偏好标签（预留）。

- 行为说明：
  - 若已配置环境变量 `AMAP_WEB_KEY`：
    - 驾车：调用 `https://restapi.amap.com/v3/direction/driving?strategy=10`，可返回多条 `paths`；
    - 步行：调用 `https://restapi.amap.com/v3/direction/walking`；
    - 公交：调用 `https://restapi.amap.com/v3/direction/transit/integrated`；当请求未提供 `city`/`citycode` 时，后端会优先以起点（失败则以终点）通过逆地理编码推断 `citycode`，策略默认 `0`；跨城场景下若起终城市不同，服务端会自动携带 `cityd`（终点城市 citycode）。
    - 返回的 `options` 解析自高德结果（`duration_min`、`distance_km`；`legs` 当前仅标注 `mode`）。
  - 未配置或调用失败：
    - 当 `mode` 为 `drive/walk` 时，优先调用 `https://restapi.amap.com/v3/distance`（`type=1` 驾车、`type=3` 步行）作为中等精度回退；
    - 若仍失败，再回退到基于 Haversine 直线距离与速度假设（驾车≈40km/h、公交≈20km/h、步行≈4.5km/h）的估算方案。

- 请求示例（驾车）：

```json
{
  "origin": {"lng": 116.4343, "lat": 39.9091},
  "destination": {"lng": 116.4344, "lat": 39.9082},
  "mode": "drive"
}
```

- 请求示例（步行）：

```json
{
  "origin": {"lng": 116.4343, "lat": 39.9091},
  "destination": {"lng": 116.4344, "lat": 39.9082},
  "mode": "walk"
}
```

- 请求示例（公交，需 `city`/`citycode`）：

```json
{
  "origin": {"lng": 116.4815, "lat": 39.9905},
  "destination": {"lng": 116.4651, "lat": 39.9995},
  "mode": "transit",
  "city": "010"
}
```

- 响应示例（节选）：

```json
{
  "options": [
    {"label": "最快", "duration_min": 23, "distance_km": 9.8, "legs": [{"mode": "transit"}]},
    {"label": "少换乘", "duration_min": 28, "distance_km": 9.5, "legs": [{"mode": "transit"}]}
  ]
}
```

#### 错误处理与配额说明（附）

- **常见错误码（AMap Web 服务）**
  - 20000 INVALID_PARAMS：参数非法
  - 20001 MISSING_REQUIRED_PARAMS：缺少必填参数
  - 20800 OUT_OF_SERVICE：点位不在中国大陆陆地范围
  - 20801 NO_ROADS_NEARBY：附近无可用道路
  - 20802 ROUTE_FAIL：路径计算失败（道路连通性等）
  - 20803 OVER_DIRECTION_RANGE：起终点距离过长
  - 300** ENGINE_RESPONSE_DATA_ERROR：服务响应失败（建议检查参数，必要时提工单）
  - 40000 QUOTA_PLAN_RUN_OUT：余额/配额耗尽
  - 40002 SERVICE_EXPIRED：服务到期
  - 40003 ABROAD_QUOTA_PLAN_RUN_OUT：海外服务余额耗尽
  - 官方参考：[`错误码说明`](https://lbs.amap.com/api/webservice/guide/tools/info/)
- **服务端处理策略（/api/route/options）**
  - 记录并透传 AMap `status/info/infocode` 到服务端日志，便于排错与容量评估。
  - 驾车/步行：主算路失败→调用 `v3/distance`（drive=type=1，walk=type=3）→仍失败则回退到 Haversine + 速度估算。
  - 公交：主算路失败→回退到 Haversine + 速度估算（占位方案）。
  - 公交 `city/citycode` 缺省：后端基于起点逆地理自动推断 `citycode`，失败再试终点。
- **配额/QPS 与运维建议**
  - 在控制台查看与管理配额/QPS：[控制台-配额管理](https://console.amap.com/dev/flow/manage)
  - 官方参考：[流量限制说明](https://lbs.amap.com/api/webservice/guide/tools/flowlevel)
  - 基础服务配额与定价说明：[定价-基础服务配额说明](https://lbs.amap.com/upgrade)
  - 建议增加短期缓存与节流日志，必要时购买/升级流量与并发额度。

#### 配置项（服务端）

- `AMAP_WEB_KEY`：高德 Web 服务 Key。配置后自动启用 AMap 接口，未配置则全走占位估算。

### 14.2 `POST /api/route/plan`

- 功能：为给定的一组景点生成“推荐游览顺序”，并返回估算总路程。
- 请求体示例：

```json
{
  "spot_ids": [3, 6, 5, 8],
  "start_spot_id": 3,
  "optimize": "2opt",
  "distance_weight": 0.7,
  "popularity_weight": 0.3
}
```

- 参数说明：
  - `spot_ids`：必填，数组且长度 ≥ 2。
  - `start_spot_id`：可选，指定起点；未提供或不在 `spot_ids` 中时默认取 `spot_ids[0]`。
  - `optimize`：可选，默认 `greedy`，可选：
    - `greedy`：最近邻启发式（快，质量一般）
    - `2opt`：先贪心再 2-opt 局部优化（更短路线）
    - `balanced`：综合距离与景点热度（适合追求体验的排序）
  - `distance_weight / popularity_weight`：仅 `balanced` 生效，默认 `0.7/0.3`。

- 响应示例（节选）：

```json
{
  "route": [
    {"id": 3, "name": "景点A", "longitude": 111.6, "latitude": 40.7},
    {"id": 6, "name": "景点B", "longitude": 111.7, "latitude": 40.8}
  ],
  "meta": {
    "total_distance_km": 12.34,
    "spot_ids_input": [3, 6, 5, 8],
    "missing_ids": [],
    "algorithm": "greedy_with_2opt_optimization",
    "optimize_mode": "2opt",
    "amap_used": false
  }
}
```

- 行为说明：
  - 若配置 `AMAP_WEB_KEY` 且外呼成功，`meta.total_distance_km` 会使用高德驾车路径规划结果；否则为 Haversine 直线距离的近似值。
  - 当部分景点缺少坐标时，会回退为尽量“保持可用”的顺序输出（不会因坐标缺失直接报 500）。

- 典型错误：
  - `spot_ids` 不是数组或长度不足：`400`，`{"error": "spot_ids must be a list with at least two ids"}`
  - `spot_ids` 含非整数：`400`，`{"error": "spot_ids must contain integers"}`
  - `start_spot_id` 非整数：`400`，`{"error": "start_spot_id must be an integer"}`
  - 有效景点不足 2 个：`400`，`{"error": "at least two scenic spots must exist"}`

### 14.3 高级参数（请求覆盖）

- 可选请求字段（便于灰度与调参）：
  - `api_version`: `v3 | v5`（覆盖服务端默认）。
  - `strategy`: 通用策略参数（传入则分别用于各模式的默认策略）。
  - `drive_strategy`: 驾车策略（v3 默认 `10`；v5 默认 `32`）。
  - `transit_strategy`: 公交策略（v3 默认 `0`；v5 支持 `0..8`，具体含义见官方文档）。
  - `city1` / `city2`（v5 公交）：城市编码（仅 `citycode`，如“010”）。

- 请求示例（v5 + 公交换乘 + 自定义策略）：

```json
{
  "origin": {"lng": 116.4815, "lat": 39.9905},
  "destination": {"lng": 116.4651, "lat": 39.9995},
  "mode": "transit",
  "api_version": "v5",
  "city1": "010",
  "city2": "010",
  "transit_strategy": "0"
}
```

- 请求示例（v5 + 驾车 + waypoints 多途经点）：

```json
{
  "origin": {"lng": 116.4343, "lat": 39.9091},
  "destination": {"lng": 116.4344, "lat": 39.9082},
  "mode": "drive",
  "api_version": "v5",
  "waypoints": [
    {"lng": 116.401, "lat": 39.901},
    [116.402, 39.902],
    "116.403,39.903"
  ]
}
```

- 说明：服务端会将 `waypoints` 序列化为 `"116.401,39.901;116.402,39.902;116.403,39.903"` 传入 v5 驾车端点。最多支持 16 个途经点，超过将返回 400。

- 请求示例（v5 + 公交 + ad1/ad2 + AlternativeRoute/multiexport/nightflag）：

```json
{
  "origin": {"lng": 116.4815, "lat": 39.9905},
  "destination": {"lng": 116.4651, "lat": 39.9995},
  "mode": "transit",

  "city1": "010",
  "city2": "010",
  "ad1": "110105",
  "ad2": "110108",

```

### 14.4 `GET /api/route/metrics`

- **功能**：返回与路线方案相关的轻量运行指标，便于观察高德调用成功/失败、回退链使用与缓存/限流效果。
- **说明**：仅为开发与运维用途；不包含任何个人数据；数值为进程级内存计数，随进程重启清零。
  - **字段补充**：响应包含 `summary` 汇总对象（便于快速观测）：
    - `amap_success_total`、`amap_fail_total`、`amap_success_rate`
    - `route_options_total`、`source_breakdown`
    - `fallback_amap_distance`、`fallback_haversine`
    - `amap_cache_hit`、`amap_cache_miss`、`amap_rate_limited_skips`
    - `infer_citycode_used`
    - `top_fail_infocode`（按 infocode 的 TopN 失败分布，如 [["30000",2],["20803",1]]）
    - `top_fail_modes`（按模式/版本的 TopN 失败分布，如 [["v5_transit",2],["v3_drive",1]]）
- **响应示例（节选）**：

```json
{
  "uptime_sec": 12345,
  "metrics": {
    "amap_version_used": {"v3": 21, "v5": 34},
    "amap_route_success": {"v3_drive": 10, "v3_walk": 3, "v3_transit": 2, "v5_drive": 9, "v5_walk": 5, "v5_transit": 6},
    "amap_route_fail": {"v3_drive": 1, "v5_transit": 2},
    "amap_fail_infocode": {"30000": 2, "20803": 1},
    "route_options_source": {"amap": 40, "amap_distance": 5, "haversine": 3},
    "fallback_amap_distance": 5,
    "fallback_haversine": 3,
    "amap_cache_hit": 120,
    "amap_cache_miss": 35,
    "amap_rate_limited_skips": 0,
    "infer_citycode_used": 8
  }
}
```

#### 运行指标与节流（说明）

- **节流策略**：
  - 每进程对 `restapi.amap.com` 施加简易限速：默认 ≤ 8 次/秒；超限请求直接跳过并记录 `amap_rate_limited_skips`。
  - 内置 TTL=60s 的内存缓存（最多 128 条），命中与未命中分别计入 `amap_cache_hit/amap_cache_miss`。
  - 计数口径：`amap_rate_limited_skips` 为在发起 HTTP 请求前因本地 QPS 限流被丢弃的次数；`amap_cache_hit` 仅在缓存条目未过期且命中时累加，`amap_cache_miss` 为未命中初始缓存的次数。
- **回退与推断**：
  - AMap 算路失败计入 `amap_route_fail`，并按模式与版本分桶（如 `v5_drive`）。
  - 启用 AMap 距离接口回退计数 `fallback_amap_distance`，最终直线估算计数 `fallback_haversine`。
  - 当 `transit` 缺省 `city`/`citycode` 时发生逆地理推断，计入 `infer_citycode_used`。
- **失败诊断**：
  - 记录 `amap_fail_infocode`（如 `20800/20802/300**/400**`），结合日志中的 `status/info/infocode` 与配额面板排查。

## 15. 偏好对话 Agent（新增）

### 15.1 `POST /api/agent/chat`

- 功能：将用户自然语言偏好转为结构化 `slots`，并按需写入用户画像（`user_profile`），同时生成行程草案预览（用于前端实时展示）。
- 适用场景：前端“我的偏好”对话式交互（Preference Chat）。

- 请求体（JSON）：
  - `user_id`：可选。登录用户传入后，后端会尝试更新画像；未登录可不传。
  - `text`：可选。单句用户输入。
  - `messages`：可选。对话历史（数组）。服务端会优先取 `messages` 中最后一条 `role="user"` 的内容作为本轮输入（若不存在则回退使用 `text`）。

请求示例：

```json
{
  "user_id": 1,
  "text": "想自驾去呼和浩特玩3天，预算3000，亲子",
  "messages": [
    {"role": "assistant", "content": "你好！请描述你的出行偏好..."},
    {"role": "user", "content": "想自驾去呼和浩特玩3天，预算3000，亲子"}
  ]
}
```

- 响应体（JSON）：
  - `reply`：助手回复（中文）。
  - `slots`：结构化槽位（可能为空对象）。
  - `actions`：动作建议（数组，可能为空），典型包含：
    - `upsert_profile`：建议写入画像
    - `suggest_itinerary`：建议生成行程草案
  - `profile`：本次更新后的画像（若未触发更新则为 `null`）。
  - `itinerary`：行程草案预览（若未生成则为 `null`）。

响应示例（节选）：

```json
{
  "reply": "已理解：目的地 呼和浩特，3天，出行方式 自驾，预算档位 2，风格 family。已为你生成行程草案。",
```

#### 行为与边界说明

- **输入长度限制**：单次输入（取最后一条用户消息）长度 > `AGENT_MAX_INPUT_CHARS`（默认 800）时，返回 `400`：`{"error": "输入过长，请简化描述后再试"}`。
- **对话轮次限制**：当 `messages` 中 `role="user"` 的回合数 > `AGENT_MAX_TURNS`（默认 20）时，返回 `400`：`{"error": "对话轮次过多，请清空对话后继续"}`。
- **天数默认与上限**：若未识别天数，默认 2 天；最终天数会被夹在 `1..10`。
- **LLM 与回退**：
  - 当服务端配置了 `LLM_API_KEY` 时，优先走 LLM 结构化抽取；
  - 未配置或 LLM 调用失败时，自动回退到本地正则规则抽取（可用但能力有限）。
- **提示注入防护（最小化）**：
  - 服务端对用户输入进行最小化转义（如反引号与尖括号），并在 LLM `system` 指令中明确“忽略与抽取无关的指令/请求”。

#### 服务端配置项（环境变量）

- `LLM_API_KEY`：大模型 API Key（不建议写入代码或提交到仓库）。
- `LLM_BASE_URL`：大模型兼容接口 Base URL，默认 `https://api.openai.com/v1`。
- `LLM_MODEL`：模型名，默认 `gpt-3.5-turbo`。
- `LLM_TIMEOUT_SECONDS`：请求超时秒数，默认 8。
- `AGENT_MAX_INPUT_CHARS`：对话单次输入最大长度（字符数），默认 800。
- `AGENT_MAX_TURNS`：对话中用户最大回合数（按 `messages` 中 `role="user"` 计数），默认 20。

## 16. 用户画像（手动偏好设置）

> 说明：用于前端“我的偏好”页面保存/读取用户显式偏好；与离线 `aggregate_user_profile.py` 的“基于评分的偏好聚合”互补。

### 16.1 `GET /api/user/profile/{user_id}`

- 功能：获取指定用户画像（若不存在则返回 `null`）。

响应示例：

```json
{
  "profile": {
    "user_id": 1,
    "cluster_id": null,
    "prefer_scenic_types": "草原,湖泊",
    "prefer_food_types": "烧烤",
    "travel_style": "family",
    "budget_level": 2,
    "travel_frequency": 12,
    "feature_vector": null,
    "updated_at": "2025-01-01T12:00:00"
  }
}
```

### 16.2 `POST /api/user/profile`

- 功能：创建或更新用户画像中的“手动偏好设置”。

请求体示例：

```json
{
  "user_id": 1,
  "travel_style": "relax",
  "budget_level": 2,
  "prefer_scenic_types": "草原,湖泊",
  "prefer_food_types": ["烧烤", "奶茶"]
}
```

行为与校验：

- `user_id` 必填且必须为整数；用户不存在时返回 `400`：`{"error": "user not found"}`。
- `travel_style` 可传 `null`/空字符串清空。
- `budget_level` 允许 `null`/空字符串清空；非空时必须为 `1/2/3`，否则返回 `400`：`{"error": "budget_level must be 1, 2, or 3"}`。
