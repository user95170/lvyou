## 1. 项目整体概述

- 项目名称：内蒙古旅游智能推荐与出行指南系统
- 核心目标：
  - 基于多源数据构建旅游资源知识库；
  - 对游客进行画像和偏好建模；
  - 提供吃、住、行、游一体化个性化推荐与出行指南；
  - 支持简单路线规划，并预留与地图/交通数据的深度融合能力。
- 当前版本定位：
  - 已实现一个端到端可运行的 Web 原型系统；
  - 具备用户注册登录、资源浏览、评分与行为记录、用户偏好设置、推荐与相似景点、行程规划等完整功能闭环；
  - 已完成多源数据示例接入（OTA 与社交媒体）和离线评估脚本设计。

> 详细架构和实施路线参考：《项目架构与实施规划.md》。

## 2. 系统功能与模块总结

### 2.1 后端功能模块

- 用户与认证：
  - 注册：校验用户名唯一性与密码复杂度，写入 `user` 表；
  - 登录：验证用户名密码，返回用户基本信息。
- 资源浏览：
  - 景点列表 `/api/scenic-spots`：支持城市、类型、关键词等筛选；
  - 酒店列表 `/api/hotels`：支持城市、星级、价格区间；
  - 美食列表 `/api/foods`：支持城市、菜系、人均价格区间。
- 行为与评分：
  - 行为日志 `/api/behaviors`：记录浏览、点击等操作；
  - 评分 `/api/ratings`：对景点/酒店/美食打分并更新聚合评分。
- 用户画像与偏好：
  - 用户画像表 `user_profile` 存储偏好类型、预算、出行风格；
  - API `/api/user/profile` 支持读取/更新画像信息。
- 推荐与相似景点：
  - 景点推荐 `/api/recommend/scenic-spots`：热门 + 画像 + 多源融合排序；
  - 酒店推荐 `/api/recommend/hotels`、美食推荐 `/api/recommend/foods`：热门排序；
  - 相似景点 `/api/recommend/scenic-spots/{spot_id}/similar`：基于 item-based CF 的邻居推荐。
- 行程规划：
  - `/api/route/plan`：按所选景点生成推荐游览顺序与总路程（最近邻启发式 + 高德 API 可选）。

### 2.2 前端功能模块

- 登录/注册页：完成用户基础身份流程；
- 首页：展示推荐景点/酒店/美食；
  - 景点卡片上展示“推荐理由”（结合内部评分、OTA、社交媒体和画像）。
- 目的地浏览页：按类型/城市浏览，支持一键评分；
  - 景点卡片提供“查看相似景点”，展示协同过滤结果。
- 我的偏好页：设置出行风格、预算、偏好类型，与后端画像接口联动；
- 行程规划页：基于城市与景点选择生成行程规划结果。

> 前端交互与路由详情参考：`frontend/src/views/*.vue` 与《接口与模块设计.md》。

## 3. 推荐算法与多源数据亮点

### 3.1 算法层面的实现

- Baseline 1：热门推荐
  - 基于内部评分与评分人数排序，作为未登录/无画像用户的默认策略。
- Baseline 2：画像驱动推荐
  - 利用 `UserProfile.prefer_scenic_types` 等字段，在候选集中优先推荐符合偏好类型的景点。
- 多源融合推荐（景点）
  - 综合使用：
    - 内部评分聚合的 `popularity_score`（`source_type = "internal_rating"`）；
    - OTA 数据中的外部评分与评论数（`source_type = "ota_stats"`（兼容历史 `ota`），由 `load_ota_data.py` 写入 `content_standard.summary`，读取时兼容 `external_rating/rating` 与 `review_count/external_review_count`）；
    - 社交媒体数据中的 `interaction_sum` 与 `sentiment_avg`（`source_type = "social_media"`，当前为骨架预留）；
  - 通过综合得分函数 `_compute_multi_source_score`，实现“内部评分 + 外部口碑 + 社交热度”的融合排序。
- 协同过滤（item-based CF）
  - 离线脚本 `aggregate_scenic_cf.py` 基于评分日志构建景点间相似度矩阵；
  - 在线接口 `/api/recommend/scenic-spots/{spot_id}/similar` 提供相似景点推荐；
  - 前端在浏览页中通过“查看相似景点”展示结果，直观体现 CF 的价值。

### 3.2 多源数据接入

- OTA 数据（当前已接入）：
  - CSV：`backend/data/ota_sample.csv`，包含 70 条来自携程、美团、去哪儿、大众点评等平台的真实景点/酒店/美食数据；
  - 采集与加载：`OTACollector`（`app/pipelines/collectors/ota_collector.py`）+ `app/pipelines/load_ota_data.py`，将外部评分和评论数等写入 `content_standard`：
    - `entity_type = scenic_spot | hotel | food_place`（兼容历史 `food`）；
    - `source_type = "ota_stats"`（兼容历史 `ota`）；
    - `summary` 中包含 `platform`、`external_rating/rating`、`review_count/external_review_count`、`price_range`、`tags` 等字段。
- 社交媒体示例数据（骨架预留）：
  - CSV：`data/social_media_scenic_sample.csv`；
  - 脚本：`app/pipelines/social_media_fetcher.py` 把热度与情感信息写入 `content_standard`（`social_media`）。

> 推荐算法与实验设计的详细说明参考：《推荐算法与实验记录.md》。

## 4. 系统优化建议（可选实施）

> 本节重点列出“有价值但不影响基本验收”的优化点，可根据时间和精力择机实现。

### 4.1 后端与数据库优化

- **查询与索引优化**：
  - 根据实际数据量，为高频条件（`city`、`type/category`、`rating_count`）增加合适索引；
  - 检查推荐与浏览接口的 SQL 执行计划，避免全表扫描。
- **分页与限制**：
  - 确保所有列表接口（景点/酒店/美食）都有合理的 `page_size` 上限，防止一次性拉取过多数据。
- **错误处理与日志**：
  - 对外部 API 调用（高德）增加超时与错误日志记录；
  - 推荐与行程规划接口在异常情况下返回明确的错误信息而非 500。

### 4.2 前端与用户体验优化

- **加载状态与提示**：
  - 对所有耗时操作（评分、相似景点、路线规划）统一使用按钮禁用与加载文本；
  - 对无数据情况提供友好提示（如“当前条件下暂无景点数据”）。
- **推荐解释性增强**：
  - 在相似景点列表中展示部分解释（如“与 X 景点在评分行为上相似”）。
- **布局与视觉优化**：
  - 统一卡片高度与字体风格，适当增加留白，提高可读性；
  - 可选引入简单的图标或颜色标签（如“热门”“高口碑”等）。

### 4.3 算法与数据层优化

- **评估指标扩展**：
  - 在 `evaluate_recommendation.py` 中增加 Recall@K、MAP、NDCG 等指标；
- **多源数据权重调参**：
  - 对多源综合得分中的权重做网格搜索或简单实验，寻找更合理的组合；
- **冷启动与长尾问题**：
  - 对评分较少的景点，增加基于内容/标签的简单推荐策略。
